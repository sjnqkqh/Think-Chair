## 0. 계획 요약 (두괄식)

**결론:** Draftsmith는 FastAPI + Jinja2 SSR 기반의 대화형 글감 발굴 에이전트다. Phase 1(MVP)에서 **회원가입/로그인 → 단일 컨셉(기술 딥다이브) → 대화 → 초고 생성 → 로컬 저장**의 골격을 완성하고, Phase 2에서 다중 컨셉·버전관리·Velog 연동을 붙이며, Phase 3에서 RAG·지표를 추가한다.

**핵심 결정 사항:**
1. **레이어드 아키텍처**: `api` (라우터) → `services` (도메인 로직) → `graph` (LangGraph 에이전트) → `repositories` (DB/스토리지) → `models` (ORM).
2. **에이전트 코어 = LangGraph**: 대화·페이즈 전환·초고/탈고 생성은 모두 **LangGraph `StateGraph`** 로 모델링. 노드=페이즈, 엣지=조건부 라우팅, 체크포인터=대화 영속성. 순수한 `LLMClient` 래퍼는 만들지 않는다.
3. **LLM 바인딩**: LangChain의 `ChatAnthropic`(Haiku 우선), `ChatDeepSeek`, `ChatGoogleGenerativeAI` 등을 LangGraph 노드 내부에서 직접 사용. 모델 스위칭은 그래프 config `configurable={"model": "..."}` 로 처리.
4. **영속성**: LangGraph `SqliteSaver`(Phase 1) → `PostgresSaver`(Phase 3+). 원고별 `thread_id = manuscript.id`.
5. **인증**: Bcrypt + JWT(HttpOnly Cookie). SSR 페이지 보호는 의존성 `require_user`.
6. **파일 스토리지**: `FileStorage` 추상 → `LocalFileStorage`(Phase 1) → `S3FileStorage`(Phase 3+).
7. **프론트엔드**: Jinja2 + HTMX + Alpine.js + Tailwind CSS. SPA 없음. LangGraph `astream` 을 SSE로 브릿지.
8. **한자 필터**: 모든 LLM 출력 노드의 후처리 훅에서 `sanitize_hanja()` 통과.

### 1.2 LangGraph 상태 그래프

```
                    ┌──────────────┐
     start ─────▶  │  router      │  ◀── user_action ∈
                    │ (조건 분기)  │      {"say","inspect","feedback",
                    └──────┬───────┘       "outline","draft","polish",
                           │                "finalize"}
        ┌──────────┬───────┼──────────┬─────────┬──────────┐
        ▼          ▼       ▼          ▼         ▼          ▼
   ┌────────┐ ┌────────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌────────┐
   │converse│ │feedback│ │outline│ │ draft │ │polish │ │finalize│
   └────┬───┘ └───┬────┘ └───┬───┘ └───┬───┘ └───┬───┘ └───┬────┘
        │        │          │         │         │         │
        └────────┴──────────┼─────────┼─────────┘         │
                            ▼         ▼                    │
                      ┌────────────────────┐              │
                      │  hanja_guard       │              │
                      │ (한자 제거 후처리) │              │
                      └──────┬─────────────┘              │
                             │                             │
                     ┌───────┴────────┐                    │
                     ▼                ▼                    ▼
              ┌────────────┐   ┌──────────────┐    ┌──────────────┐
              │ (converse) │   │persist_version│    │ finalize DB  │
              │   END      │   │ (파일+DB row)│    │  update      │
              └────────────┘   └──────┬───────┘    └──────┬───────┘
                                      │                    │
                                      ▼                    ▼
                                     END                  END
```

- **State**: `DraftsmithState(TypedDict)` — `messages: Annotated[list, add_messages]`, `manuscript_id`, `concept`, `topic`, `audience_level`, `user_action`, `pending_version`.
- **Checkpointer**: `SqliteSaver.from_conn_string("draftsmith_ckpt.db")` — `thread_id=str(manuscript.id)` 로 원고별 완전 격리.
- **모델 스위칭**: `graph.invoke(state, config={"configurable": {"thread_id": ..., "model": "haiku"}})`. `llm_registry.get(name)` 이 `ChatAnthropic` 등을 반환.

### 1.3 초고/탈고 흐름 (LangGraph 관점)

```
[HTMX POST /api/generate/draft?manuscript_id=X]
    │
    ▼
[chat_service.run_action(manuscript_id=X, action="draft")]
    │  state = {"user_action": "draft"}
    │  config = {"configurable": {"thread_id": X, "model": "haiku"}}
    ▼
[graph.ainvoke(state, config)]
    │  router → draft 노드 → hanja_guard → persist_version → END
    │  persist_version 노드가 FileStorage.save + ManuscriptVersion INSERT
    ▼
[반환된 최종 State에서 pending_version 추출]
    │
    ▼
[Jinja partial (우측 버전 목록) 렌더 → HTMX OOB swap]
```

class ConceptType(str, Enum):
    TECH_DEEPDIVE = "tech_deepdive"
    RETROSPECTIVE = "retrospective"
    ESSAY = "essay"
    TIL = "til"
    TEACHING = "teaching"

class ManuscriptStatus(str, Enum):
    DRAFTING = "drafting"        # 대화 진행 중

    POLISHING = "polishing"      # 탈고 중

    id: Mapped[UUID] = mapped_column(primary_key=True, default=uuid4)

    kind: Mapped[str]                          # "draft" | "polish"
    revision: Mapped[int]                      # 1, 2, 3...
    storage_key: Mapped[str]                   # ~/storage/{kind}s/{uuid}.md

    created_at: Mapped[datetime] = mapped_column(default=datetime.utcnow)

## 5. LangGraph 에이전트 상세 설계

### 5.0 상태 정의 (`app/graph/state.py`)

```python
from typing import Annotated, Literal, TypedDict
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

UserAction = Literal["say", "inspect", "feedback", "outline", "draft", "polish", "finalize"]

class PendingVersion(TypedDict, total=False):
    kind: Literal["outline", "draft", "polish"]
    content: str
    storage_key: str          # persist_version 노드가 채움
    version_id: str

class DraftsmithState(TypedDict):
    manuscript_id: str
    concept: str              # ConceptType value
    topic: str
    audience_level: str | None
    user_action: UserAction
    messages: Annotated[list[BaseMessage], add_messages]
    pending_version: PendingVersion | None
```

### 5.1 그래프 빌더 (`app/graph/builder.py`)

```python
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.sqlite import SqliteSaver

def build_graph(checkpointer: SqliteSaver):
    g = StateGraph(DraftsmithState)
    g.add_node("router", router_node)
    g.add_node("converse", converse_node)
    g.add_node("feedback", feedback_node)
    g.add_node("outline", outline_node)
    g.add_node("draft", draft_node)
    g.add_node("polish", polish_node)
    g.add_node("finalize", finalize_node)
    g.add_node("hanja_guard", hanja_guard_node)
    g.add_node("persist_version", persist_version_node)

    g.set_entry_point("router")

    g.add_conditional_edges("router", route_by_action, {
        "say": "converse", "inspect": "converse",
        "feedback": "feedback", "outline": "outline",
        "draft": "draft", "polish": "polish",
        "finalize": "finalize",
    })

    # 대화형 노드 → 한자 필터 → END
    for n in ("converse", "feedback"):
        g.add_edge(n, "hanja_guard")

    # 산출물 노드 → 한자 필터 → persist_version → END
    for n in ("outline", "draft", "polish"):
        g.add_edge(n, "hanja_guard")

    g.add_conditional_edges("hanja_guard",
        lambda s: "persist_version" if s.get("pending_version") else END,
        {"persist_version": "persist_version", END: END})
    g.add_edge("persist_version", END)
    g.add_edge("finalize", END)

    return g.compile(checkpointer=checkpointer)
```

### 5.4 한자 가드 노드 (`app/graph/nodes/hanja_guard.py`)

```python
from app.core.hanja_filter import sanitize_hanja
from langchain_core.messages import AIMessage

def hanja_guard_node(state: DraftsmithState) -> dict:
    patch: dict = {}
    last = state["messages"][-1] if state["messages"] else None
    if isinstance(last, AIMessage) and last.content:
        clean = sanitize_hanja(last.content)
        if clean != last.content:
            patch["messages"] = [AIMessage(content=clean, id=last.id)]  # 동일 id로 대체
    pv = state.get("pending_version")
    if pv and pv.get("content"):
        pv = {**pv, "content": sanitize_hanja(pv["content"])}
        patch["pending_version"] = pv
    return patch
```

### 5.8 프롬프트 조합 방식

```
Final system prompt = base_persona + global_constraints(hanja/ascii/emoji ban)
                       + concept_content(phase 역할에 따라 선택)
                       + phase_instruction + topic/audience context
```

**concept_content 선택 규칙 (신규):** 초고/탈고 "생성"에 쓰는 프롬프트와 "점검"에 쓰는 프롬프트는 서로 다르다. 하나의 짧은 concept_policy를 모든 phase에 재사용하지 않고, phase를 아래 3개 역할군으로 분류해 concept마다 다른 콘텐츠를 선택한다.

| phase 역할군 | 대상 phase | 사용하는 concept 콘텐츠 | 비고 |
|---|---|---|---|
| CONVERSATION | say | `purpose` (generate.md의 "## 목적" 섹션만 추출) | 대화 맥락 힌트 정도의 짧은 텍스트 |
| REVIEW (점검) | inspect, feedback | `checkpoint` (`checkpoint.md` 원문) | 경량 체크리스트, 대화 중 반복 호출되므로 짧게 유지 |
| GENERATION (생성) | outline, draft, polish | `generate` (`generate.md` 원문) | 전체 생성 기준 원문, 생성 시 1회성 호출이므로 길어도 무방 |

- **5개 컨셉 모두** `generate.md`/`checkpoint.md`를 보유한다 (`prompt-sample/`에서 전달받음 — tech_deepdive/essay/retrospective/til/teaching 전부). 폴백(policy) 로직은 더 이상 필요 없다.
- `get_concept_content(concept, phase) -> PromptTemplate`가 위 표의 역할 매핑(`PHASE_ROLES = {"say": "purpose", "inspect": "checkpoint", "feedback": "checkpoint", "outline": "generate", "draft": "generate", "polish": "generate"}`)을 내부에서 처리하므로, `build_system_prompt`나 그래프 노드는 컨셉/phase 조합을 직접 분기할 필요가 없다.
- `generate.md`/`checkpoint.md`는 file 기반으로 유지(추후 DB 이관 예정, 지금은 아님). 로딩된 원문은 `PromptTemplate(text=원문 전체, used_when=..., description=...)`로 감싸 다른 템플릿과 동일한 인터페이스를 유지한다. `purpose`는 `md_loader.extract_section(generate_md, "목적")`로 파생되므로 별도 원본 없이 `generate.md`를 단일 소스로 유지한다.

### 5.3 concept 콘텐츠 예시 (concepts/tech_deepdive/)

컨셉마다 짧은 `policy` 텍스트 하나가 아니라, `prompt-sample/`에서 전달받은 **역할별 원문 문서 2개**를 그대로 사용한다 (5개 컨셉 모두 동일 패턴):

## 10. 실행 규칙 (Sonnet 준수 사항)

1. 각 태스크 완료 후 반드시 `uv run pytest -x`와 `uv run ruff check .` 실행. 실패 시 다음 태스크로 넘어가지 않는다.
2. **LLM 호출은 반드시 LangGraph 노드 안에서 수행한다.** FastAPI 라우터나 서비스에서 `ChatAnthropic.invoke`를 직접 호출하는 것을 금지한다. 신규 페이즈가 필요하면 노드를 추가하고 라우터 엣지를 확장한다.
3. **직접적인 `LLMClient` 스타일의 자체 래퍼 클래스를 만들지 않는다.** 모델 교체가 필요하면 `llm_registry` 에 LangChain 채팅 모델을 등록하는 방식으로만 확장한다.
4. LLM 응답 저장·표시 경로는 그래프 내부 `hanja_guard` 노드를 반드시 거친다. 라우터가 그래프를 우회해 직접 응답을 만들 수 없다.
5. ASCII 표/텍스트 그림 금지 규칙을 프롬프트에서 제거하지 않는다.
6. `Manuscript.concept`은 생성 이후 update 라우트를 만들지 않는다(변경 불가).
7. 파일 저장 키는 `f"{kind}s/{uuid4()}.md"` 규약을 지키며, 하드코딩된 사용자 홈 경로를 코드에 직접 쓰지 않는다(`settings.storage_root` 만 사용).
8. 프론트엔드는 별도 SPA를 추가하지 않는다. HTMX partial + Alpine store 로 상태 관리.
9. 대화 이력은 SQLAlchemy `ChatMessage` 테이블과 LangGraph `SqliteSaver` 체크포인트 **두 곳에 이중 저장**한다. 전자는 UI 렌더링용, 후자는 그래프 재시작·부분 재실행용.
10. 커밋 단위: 태스크 1개 = 커밋 1개. 커밋 메시지 규칙: `feat(scope): ...`, `test(scope): ...`, `chore: ...`.
