## 1. 유닉스 환경에서 개인 프로젝트 서버를 실행시킨 뒤, 핵심 명령어로 프로세스, 스레드, 메모리 상태를 조회하고 그 동작을 분석한 보고서를 작성해 보세요. 

### 1. 실행 환경 및 서버 구동

AWS EC2의 Linux 환경에서 FastAPI 기반 개인 프로젝트 서버를 실행하였다. Python 의존성 및 실행 환경 관리는 `uv`를 사용하였다.

```bash
# uv 설치
curl -Lsf https://astral.sh/uv/install.sh | sh
source ~/.bashrc

# 프로젝트 의존성 설치
uv sync

# Uvicorn 서버를 백그라운드로 실행하고 PID 저장
nohup uv run uvicorn app.main:app --host 0.0.0.0 --port 8000 > app.log 2>&1 &
echo $! > app.pid
```

`uv sync`는 프로젝트 설정 파일을 기준으로 필요한 Python 패키지를 설치한다. 이후 `uv run uvicorn` 명령으로 FastAPI 애플리케이션을 8000번 포트에서 실행하였다. `nohup`과 `&`를 사용하여 SSH 연결이 종료되어도 서버가 계속 실행되도록 하였고, 실행 PID는 `app.pid` 파일에 저장하였다.

### 2. 성능 모니터링 및 부하 발생 방법

두 개의 터미널을 사용하였다. 첫 번째 터미널에서는 Uvicorn 프로세스의 스레드 및 CPU 상태를 관찰하였다.

```bash
top -H -p "$(cat app.pid)"
```

`-p` 옵션은 저장한 PID의 프로세스만 표시하며, `-H` 옵션은 해당 프로세스 내부의 스레드별 상태를 보여 준다. 이를 통해 스레드 수, 각 스레드의 상태, CPU 사용량을 실시간으로 확인하였다.

두 번째 터미널에서는 `curl`을 이용해 서버 내부에서 로컬 FastAPI 서버로 HTTP 요청을 전송하였다.

```bash
# 서버의 기본 응답 확인
curl http://127.0.0.1:8000/

# 총 300개의 요청을 전송하며, 최대 30개를 동시에 실행
seq 1 300 | xargs -P 30 -I {} \
  curl -s -o /dev/null http://127.0.0.1:8000/
```

부하 명령은 총 300개의 요청을 만들고, 최대 30개의 `curl` 요청을 동시에 실행한다. `-s -o /dev/null` 옵션은 응답 본문을 출력하지 않아 요청 결과 화면이 과도하게 출력되는 것을 방지한다.

### 3. 관찰 결과

유휴 상태에서 Uvicorn 프로세스는 총 2개의 스레드로 구성되어 있었으며, 실행 중인 스레드는 0개, 대기 상태인 스레드는 2개로 나타났다. CPU 상태는 사용자 영역 사용률이 0.2%이고 유휴 비율은 99.8%였다. 즉, 요청이 없는 상황에서 서버는 네트워크 요청을 기다리며 CPU를 거의 사용하지 않았다.

동시 HTTP 요청을 전송한 동안 CPU 상태는 다음과 같이 변하였다.

| 항목 | 유휴 상태 | 부하 상태 |
|---|---:|---:|
| 사용자 영역 CPU (`us`) | 0.2% | 8.9% |
| 커널 영역 CPU (`sy`) | 0.0% | 7.1% |
| 소프트웨어 인터럽트 (`si`) | 0.0% | 0.3% |
| CPU 유휴 비율 (`id`) | 99.8% | 83.5% |

부하 상태에서는 사용자 영역 CPU 사용률이 8.9%로 증가하였다. 이는 FastAPI/Uvicorn의 요청 처리와 `curl` 클라이언트 프로그램의 실행이 발생했기 때문이다. 또한 커널 영역 CPU 사용률이 7.1%로 증가했는데, 이는 TCP 소켓 통신, 네트워크 패킷 처리, 시스템 호출 등 운영체제 커널의 작업이 증가한 결과로 해석할 수 있다. 소프트웨어 인터럽트도 0.3% 관찰되어 네트워크 패킷 처리 활동이 있었음을 확인하였다.

CPU 유휴 비율은 부하 상황에서도 83.5%로 유지되었다. 따라서 이번 부하는 서버의 CPU 자원을 일부 사용하게 했지만, 인스턴스의 CPU 처리 능력을 포화시키지는 않았으며 안정적으로 요청을 처리한 것으로 판단할 수 있다.

### 4. 결론

FastAPI/Uvicorn 서버는 유휴 상태에서 2개의 스레드가 HTTP 요청을 기다리며 매우 낮은 CPU 사용률을 보였다. 이후 `curl`을 통해 동시 HTTP 요청을 전송하자 사용자 영역과 커널 영역의 CPU 사용률이 증가하였다. 그러나 CPU 유휴 비율이 83.5%로 충분히 유지되어, 관찰한 부하 수준에서는 서버가 안정적으로 동작함을 확인하였다.

## 2. WireShark로 개인 프로젝트 서버의 HTTP/HTTPS 통신을 캡처해보세요.


### 1. 실습 목적

EC2에 배포한 FastAPI 서버와 외부 클라이언트 간의 HTTP 통신을 캡처하고, TCP 연결 수립부터 HTTP 요청·응답, 연결 종료까지의 과정을 분석하였다.

서버는 AWS EC2 인스턴스에서 실행하였고, 클라이언트는 별도의 맥북에서 실행하였다. 따라서 클라이언트와 서버는 서로 다른 컴퓨터 환경에 구성되었다.

### 2. 네트워크 및 보안 그룹 설정

FastAPI/Uvicorn 서버는 TCP 8000번 포트에서 실행하였다.

```bash
nohup uv run uvicorn app.main:app --host 0.0.0.0 --port 8000 > app.log 2>&1 &
echo $! > app.pid
```

초기에는 EC2 보안 그룹에서 `HTTP` 유형을 선택하여 TCP 80번 포트만 허용하였다. 그러나 서버는 8000번 포트에서 실행 중이므로 외부 클라이언트의 접속이 실패하였다.

이를 해결하기 위해 보안 그룹 인바운드 규칙을 다음과 같이 수정하였다.

| 항목 | 설정 |
|---|---|
| 유형 | 사용자 지정 TCP |
| 포트 범위 | 8000 |
| 소스 | 클라이언트 공인 IP `/32` |

HTTP는 80번 포트에서만 동작하는 것이 아니라, TCP 8000번 포트에서도 사용할 수 있다. 보안 그룹 포트를 Uvicorn 실행 포트와 동일하게 설정한 뒤 외부 접속이 정상적으로 이루어졌다.

### 3. 패킷 캡처 방법

EC2는 Amazon Linux 환경이므로 Wireshark의 CLI 도구인 Tshark를 설치하였다.

```bash
sudo dnf install -y wireshark-cli
```

서버에서 TCP 8000번 포트 통신만 캡처하도록 Tshark를 실행하였다.

```bash
sudo tshark -i any -f "tcp port 8000" \
  -w /tmp/fastapi-http.pcapng
```

Tshark 실행 중 Mac 클라이언트에서 EC2 서버로 HTTP 요청을 전송하였다.

```bash
curl -i http://13.124.73.48:8000/
```

캡처를 종료한 뒤 저장된 패킷 파일을 읽어 결과를 확인하였다.

```bash
tshark -r fastapi-http.pcapng
```

### 4. 캡처 결과

캡처 결과는 다음과 같다.

```text
1   211.244.225.211 → 172.31.35.163  TCP   54851 → 8000 [SYN]
2   172.31.35.163 → 211.244.225.211  TCP   8000 → 54851 [SYN, ACK]
3   211.244.225.211 → 172.31.35.163  TCP   54851 → 8000 [ACK]
4   211.244.225.211 → 172.31.35.163  HTTP  GET / HTTP/1.1
6   172.31.35.163 → 211.244.225.211  HTTP  HTTP/1.1 307 Temporary Redirect
8~10 TCP FIN, ACK 패킷 교환
```

| 구간 | 관찰 내용 | 분석 |
|---|---|---|
| 패킷 1~3 | SYN, SYN/ACK, ACK | TCP 3-way handshake를 통해 클라이언트와 서버의 연결이 정상적으로 수립되었다. |
| 패킷 4 | `GET / HTTP/1.1` | Mac 클라이언트가 FastAPI 서버의 루트 경로로 HTTP GET 요청을 전송하였다. |
| 패킷 6 | `HTTP/1.1 307 Temporary Redirect` | 서버는 루트 경로 요청에 대해 다른 경로로 이동하도록 리다이렉트 응답을 반환하였다. 이는 서버 오류가 아닌 정상적인 HTTP 응답이다. |
| 패킷 8~10 | FIN, ACK | 응답 처리 후 TCP 연결이 정상적으로 종료되었다. |

서버 측에서 캡처했기 때문에 EC2 서버 주소는 공인 IP가 아닌 VPC 내부 사설 IP인 `172.31.35.163`으로 나타났다. 외부 클라이언트의 공인 IP `211.244.225.211`에서 EC2 서버의 TCP 8000번 포트로 통신이 전달된 것을 확인하였다.

### 5. 결론

외부 클라이언트 (교육장 MacBook)와 EC2 FastAPI 서버 간의 HTTP 통신을 Tshark로 캡처하였다. 분석 결과 TCP 연결 수립, HTTP GET 요청, FastAPI 서버의 307 리다이렉트 응답, TCP 연결 종료 과정이 정상적으로 확인되었다.

또한 서버의 실행 포트와 AWS 보안 그룹 인바운드 포트가 일치해야 외부 접속이 가능함을 확인하였다. 초기에는 HTTP 기본 포트인 80번 포트만 허용하여 연결에 실패했지만, 서버 실행 포트인 8000번 포트를 클라이언트 공인 IP에 대해서만 허용하도록 수정한 뒤 정상적으로 통신할 수 있었다.
