# RAG 통합 프로젝트 — 과제 후기

## 두 줄 요약

**"AI 코딩에 지속적인 코드 검수를 곁들였는데도, 내가 모르는 구현과 용납 못할 퀄리티의 코드가 상당수 생성되었습니다."**  
그리고 **프로젝트의 목적이 명확한 구현체(목적 프로젝트)** 를 만드는 방식이, 단순히 기술 스택을 적용하는 과제 수행보다 훨씬 효과적이었습니다.

---

## 1. 충격 — AI 코딩의 현실

이번 프로젝트는 AI 코딩 어시스턴트(Gemini 3.5 Flash)에 구현을 전적으로 맡기고, 사람이 지속적으로 코드를 검수하면서 진행하는 방식으로 작업했습니다.  
검수는 꽤 꼼꼼하게 했다고 생각했습니다. PR 단위로 diff를 보고, 로직 흐름을 따라가고, 테스트도 돌려봤습니다.

그런데도 **프로덕트에 포함된 채로 넘어간, 내가 전혀 인지하지 못한 구현 사항들**이 꽤 있었습니다.  
그리고 그것들을 직접 수정하려고 보니 **평소 제 기준이라면 절대 용납하지 않았을 코드 퀄리티**였습니다.  
"어, 이렇게 짰다고?" 싶은 부분이 여럿 보였습니다. 지속적인 코드 검수에도 이걸 다 걸러내지 못했다는 게 충격이었습니다.

### 특히 두드러진 패턴

이 과정에서 알렉스의 조언대로 **AI가 생성한 모든 코드를 이해하려고 시도**했는데, 이 과정에서 **파이썬 생태계에 아직 익숙하지 않은 부분**이 여럿 드러났습니다.  
생각보다 낯선 코드 패턴을 자주 마주했고, 문맥적으로는 이해가 가지만 "정확히 이게 무슨 동작을 하는 코드인가" 싶은 부분도 적지 않았습니다.

예를 들면:

- SQLAlchemy의 세션 관리와 지연 로딩(Lazy Loading)
- `asyncio.gather`와 `asyncio.to_thread`의 차이점

이런 부분을 검색이나 AI를 통해 이해하고 넘어가는 시도를 하고 있으며, 이 **메타인지 자체가 중요한 수확**이었다고 생각합니다.  
내가 무엇을 모르는지 아는 것이, 앞으로의 학습 방향을 설정하는 데 큰 도움이 되고 있습니다.

### 특히 두드러진 AI 코딩 패턴

| 패턴             | 설명                                                                                                    |
|----------------|-------------------------------------------------------------------------------------------------------|
| **에러 땜빵**      | 오류가 발생했을 때, 원인을 해결하지 않고 `try-except`로 조용히 삼키거나 기본값으로 대체해버림                                            |
| **가짜 해결**      | 문제 해결을 지시하면 "수정했습니다"라고 말하지만 실제로는 에러가 그대로인 채로 넘어감                                                      |
| **비건설적 접근**    | 문제 해결 시도가 근본 원인 분석보다는 "다른 방식으로 우회"에 집중됨                                                               |
| **레이어 분리 인색**  | 서비스 레이어로 분리해야 할 작업임에도 엔드포인트에서 한꺼번에 처리하려는 경향이 매우 강함. 이미 구조가 분리된 상황에서도 명시적으로 지시하지 않으면 레이어 추가를 스스로 하지 않음 |
| **노베이스 코드 취약** | 프로젝트 구조가 아직 명확히 잡히지 않은 초기 단계에서 생성되는 코드의 질이 특히 낮았음 — 지향점이 뚜렷하지 않은 코드베이스에서는 퀄리티가 바닥을 칠 수 있다는 것을 확인      |

## 2. 환경 제약과 우회 전략

### Gemini API 쿼터 문제

처음에는 Gemini 2.5 Flash를 코딩 베이스 모델로 사용하는 것을 추천받았으나, **무료 API 쿼터가 너무 적어서** 실질적인 작업에 사용하는 것은 불가능했습니다.  
또한 Gemini Embedding API는 **분당 호출 제한이 약 100회**로 매우 낮아, 다양한 청킹 전략을 테스트하는 데 큰 제약이 있었습니다.

이에 따라 다음과 같은 우회 전략이 필요했습니다:

- **지수 백오프(Exponential Backoff)** — 재시도 로직 구현으로 API 제한 우회
- **명시적 딜레이** — 요청 간 간격을 강제로 확보
- **임베딩 비동기화** — 순차 처리 대신 백그라운드 태스크로 Embedding 수행

### DeepSeek V4 Flash — 비용 효율성

반면 평가 단계에서 사용한 **DeepSeek V4 Flash는 토큰당 비용이 매우 낮아** 부담 없이 사용할 수 있었습니다.  
RAG 평가와 증강 생성에는 생각보다 많은 API 호출이 필요하기 때문에, 이 점은 프로젝트 진행에 큰 도움이 되었습니다.

## 3. 해결의 실마리

위에서 언급한 패턴 중 가장 치명적이었던 것은 **LangChain과 RAGAS 라이브러리 간 버전 호환성 문제**였습니다.  
AI는 이 문제를 계속 우회하려고만 했고, 진짜 해결을 못했습니다.

결국 다음과 같은 방식으로 해결했습니다:

> **Opus(Claude)를 통한 문제 탐색 + 웹 서치로 유사 사례 조사 → `langchain-community` 버전 다운그레이드**
즉, 코딩에 사용한 모델(Gemini Flash)이 아닌 **더 강력한 추론 모델로 진단 + 실제 사례 리서치**를 병행해야 해결할 수 있었습니다.  

모델에 따른 성능 차이는 반드시 존재하겠지만, 보편적 모델의 코딩 성능은 지시에 따를 때 그 효과를 발휘하는 점을 다시 확인했습니다.
그리고 Opus는 항상 만족스러운 성적을 냅니다. 역시 사람이든, 모델이든 한 가지를 잘 하면 대우를 받습니다.

## 4. 좋았던 점 — 목적 프로젝트 방식

이번 과제에서 가장 의미 있었던 결정은 **과제 방식을 바꾼 것**입니다.

| 기존 방식                     | 변경한 방식                                   |
|---------------------------|------------------------------------------|
| 과제가 지시하는 기술 스택을 그냥 적용     | **해당 기술이 적용된 목적 프로젝트를 구현**               |
| "RAGAS를 써보세요" → 평가 코드만 추가 | "RAG 시스템을 만들고, 그 안에서 RAGAS로 청킹 전략을 비교하자" |
| 수동적 과제 수행                 | 적극적인 프로젝트 구현                             |

단순히 "이 기술을 써보는 것"이 아니라, **"이 기술이 필요해서 어쩔 수 없이 쓰는"** 구조로 바꾸니 훨씬 능동적으로 작업에 임할 수 있었습니다.  
아직 갈 길이 멀지만, 이 방향성은 앞으로도 유지할 생각입니다.

## 6. 추후 과제 — 골든 테스트셋 활용

골든 테스트셋(Golden Dataset)을 추가하여 RAG 평가의 정량적 기준을 마련해두었습니다.  
하지만 **아직 이 데이터셋을 본격적으로 활용한 평가는 수행하지 못한 상태**입니다.
RAG 평가 자체가 다수의 API 호출을 필요로 하는 작업인데, 현재 환경에서 이를 체계적으로 운영할 수 있는지 불확실하기 때문입니다.  
테스트셋을 실제 평가 파이프라인에 통합하고 정기적으로 실행하는 것은 다음 단계의 과제로 남겨두었습니다.

## 7. 교훈

1. **AI 코드를 검수한다는 착각에 빠지지 말자** — 꼼꼼히 본다고 생각해도 놓치는 부분이 많습니다. 특히 넓은 범위의 작업을 빠르게 진행할 때 주요한 비즈니스 로직의 변경 사항은 확인하고, 지시하지 않은 변경
   사항에 대한 검수는 누락하는 경우가 있는데 이 부분도 꼼꼼히 확인해야겠다는 교훈을 얻었습니다.
2. **프로젝트의 목적이 명확해야 코드 퀄리티도 올라갑니다** — "왜 이 기술을 쓰는가"가 분명할수록 AI도 더 나은 코드를 생성합니다.
3. **의존성 관리에 각별히 주의해야 합니다** — 라이브러리 간 버전 호환성 문제는 AI가 스스로 해결하기 가장 어려운 영역 중 하나입니다. 최신 버전이 항상 정답이 아닙니다. 또한 AI가 잘못된 루프에 빠지기
   쉬운 부분이므로 발생 징후가 보이면 개발자가 적극적으로 개입해야 할 것으로 보입니다.
