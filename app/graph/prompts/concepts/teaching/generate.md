# 수업자료 문서 품질 기준

> 이 문서는 수업자료(Teaching Material) 포스팅이 갖춰야 할 최소 품질 기준을 정의합니다.
> 모델이 중간 생성 결과물을 검토할 때, 아래 각 항목을 기준으로 현재까지 작성된 내용이 조건에 부합하는지 판단하는 용도로 사용됩니다.
> 각 기준은 "왜 중요한가" → "구체적 기준" → "위반 시 문제점" 순서로 설명합니다.

---

## 목적
학습자가 특정 개념이나 기술을 처음부터 끝까지 스스로 익힐 수 있도록 설계된 교육용 문서. 정확한 개념 설명, 실행 가능한 예제, 직접 해보는 실습을 통해 학습 목표 달성을 돕는 것이 목적.

## 0. 제목 헤딩

**기준:** 문서의 가장 위에는 세션 제목을 `#` H1 헤딩으로 지정해야 한다.

**검토 시 확인할 점:**
- 문서 첫 줄이 `# {세션 제목}` 형식의 H1 헤딩인가
- 본문 내 다른 헤딩이 H1(`#`)을 중복 사용하지 않는가 (본문은 `##` 이하 사용)

## 1. 학습 목표

수업자료는 독자가 무엇을 할 수 있게 되는지를 명확히 정의해야 한다. 목표 없는 수업은 방향 없는 여행과 같다.

### 1.1 목표의 구체성

**기준:** 학습 목표는 구체적인 행동 동사(설명한다/구현한다/분석한다)로 작성되어야 하며, 읽고 나서 무엇을 할 수 있는지 명시해야 한다.

| 나쁨 | 좋음 |
|------|------|
| "트랜스포머 이해하기" | "트랜스포머의 어텐션 수식을 한 줄씩 설명할 수 있고, 직접 구현할 수 있다" |
| "FastAPI 배우기" | "FastAPI로 CRUD API 4개를 작성하고, Swagger UI로 테스트할 수 있다" |
| "데이터베이스 개념" | "정규화 1~3정규형을 적용해서 기존 테이블을 분해할 수 있다" |

**검토 시 확인할 점:**
- "이해한다/배운다/알게 된다" 같은 모호한 동사가 아닌가
- 학습자가 수업 종료 후 "무엇을 할 수 있는가"가 명시되었는가
- 목표가 측정 가능한가 (수행 결과를 평가할 수 있는가)

### 1.2 목표 도달 확인 장치

**기준:** 학습 목표에 도달했는지 확인할 수 있는 방법(퀴즈, 실습 과제, 체크리스트)이 포함되어야 한다.

| 수준 | 확인 방법 | 예시 |
|------|----------|------|
| 기억 | 퀴즈 | "다음 중 어텐션 헤드의 역할은?" |
| 이해 | 설명 요청 | "nn.Linear와 nn.Conv1d의 차이를 설명해보세요" |
| 적용 | 실습 과제 | "주어진 데이터로 CRUD API를 작성하세요" |
| 분석 | 비교 과제 | "두 모델의 출력을 비교하고 차이를 분석하세요" |

**검토 시 확인할 점:**
- 학습 목표와 확인 방법이 일치하는가 (이해가 목표인데 암기 확인만 하면 안 됨)
- 학습자가 스스로 체크할 수 있는 형태인가
- 각 섹션 말미에 간단한 확인 질문이 있는가

## 2. 개념 설명

수업자료의 핵심은 개념을 정확하고 이해하기 쉽게 전달하는 것이다.

### 2.1 정확성

**기준:** 모든 개념 설명은 기술적으로 정확해야 한다. 비유는 이해를 돕는 보조 도구일 뿐, 정확성을 희생해서는 안 된다.

| 위반 사례 | 문제점 |
|---------|--------|
| "인터넷은 하늘에 있는 구름이다" | 비유가 지나쳐 실제 동작 원리와 괴리 발생 |
| "그냥 이렇게 하면 된다" | 원리 설명 없이 블랙박스로 만듦 |
| "대략 이런 뜻이다" | 모호한 설명, 학습자에게 잘못된 개념 심어줌 |

**검토 시 확인할 점:**
- 설명에 기술적 오류가 없는가
- 비유가 개념의 본질을 정확히 전달하는가 (비유에 치우쳐 본질이 왜곡되지 않았는가)
- "대충/그냥/아마" 같은 모호한 표현이 설명을 대체하고 있지 않은가

### 2.2 난이도 계층

**기준:** 설명은 독자 수준에 맞게 계층화되어야 한다. 모든 내용을 같은 난이도로 설명하지 않는다.

| 레벨 | 대상 | 설명 방식 |
|------|------|----------|
| **L1** | 완전 초심자 | 비유와 그림 위주, 코드 없음 |
| **L2** | 기본 문법 아는 사람 | 간단한 코드 예제, 주요 개념 설명 |
| **L3** | 경험자 | 실제 사용 패턴, Best practice, 주의사항 |
| **L4** | 심화 학습자 | 내부 동작 원리, 성능 비교, 대안 분석 |

**예시 (동일 개념, 난이도별):**
| 레벨 | "파이썬 데코레이터" 설명 |
|------|------------------------|
| L1 | "함수에 추가 기능을 붙이는 스탬프 같은 것" |
| L2 | "@log_time 같은 데코레이터로 함수 실행 시간 측정" |
| L3 | "여러 데코레이터 중첩 시 실행 순서: 위→아래 정의, 아래→위 실행" |
| L4 | "데코레이터는 클로저와 functools.wraps를 활용한 디자인 패턴" |

**검토 시 확인할 점:**
- 모든 내용이 같은 난이도로 쓰이지 않았는가
- 핵심 개념(학습 목표와 직접 연결)은 충분히 자세히 설명되었는가
- 부차적 개념은 너무 깊게 파고들지 않았는가

### 2.3 용어 설명

**기준:** 새로운 용어는 반드시 첫 등장 시 (한글+영문) 병기하고, 의미를 설명해야 한다.

**예:**
- "파이토치의 `DataLoader`(데이터 로더)는 배치 단위로 데이터를 로딩하는 유틸리티다"
- "어텐션(Attention) 메커니즘은 입력 시퀀스의 각 위치에 가중치를 부여하는 방식이다"

**검토 시 확인할 점:**
- 모든 신규 용어가 첫 등장 시 설명되었는가
- 용어가 설명 없이 갑자기 등장하는 구간이 있는가
- 영문과 한글이 일관되게 병기되었는가 (처음만 병기, 이후 통일)

## 3. 예제와 실습

수업자료에서 예제는 단순한 부록이 아니라 핵심 교수 도구다.

### 3.1 실행 가능한 코드

**기준:** 모든 코드 예제는 `import`부터 실행까지 복사+붙여넣기로 동작해야 한다.

**검토 시 확인할 점:**
- 첫 줄이 `import ...`로 시작해서 결과를 출력하거나 반환하는가
- 필요한 의존성(pip install)이 명시되었는가
- 누락된 import, 정의되지 않은 변수, 오타가 없는가
- 코드에 의존하는 외부 파일/데이터가 있다면 생성 또는 다운로드 코드도 포함되었는가

### 3.2 점진적 난이도

**기준:** 예제는 가장 단순한 형태에서 시작해서 점진적으로 복잡해져야 한다.

| 단계 | 예: FastAPI 튜토리얼 |
|------|---------------------|
| 1단계 | "Hello, World" — 가장 단순한 GET 엔드포인트 |
| 2단계 | 경로 매개변수, 쿼리 매개변수 추가 |
| 3단계 | POST 요청과 Request Body (Pydantic 모델) |
| 4단계 | 데이터베이스 연동 (SQLite) |
| 5단계 | 에러 처리, 의존성 주입 |

**검토 시 확인할 점:**
- 첫 번째 예제가 독자가 즉시 실행 가능한가 (설정/준비 없이)
- 각 단계가 이전 단계의 이해를 전제로 하는가
- 난이도 상승 폭이 적절한가 (너무 가파르지 않은가)

### 3.3 실습 과제

**기준:** 학습자가 직접 해볼 수 있는 실습 과제가 최소 1개 이상 포함되어야 한다.

**좋은 실습의 조건:**
- 학습 목표와 직접 연결될 것
- 정답이 하나가 아니라 여러 접근법이 가능할 것 (단순 암기 방지)
- 학습자가 스스로 검증할 수 있는 기준이 제공될 것
- 실습 소요 시간이 명시될 것

**검토 시 확인할 점:**
- 실습 과제가 단순한 복사 붙여넣기가 아닌가 (변형/응용이 필요한가)
- 실습 난이도가 수업자료의 독자 수준에 맞는가
- 실습 시간이 현실적인가

### 3.4 오답/오류 포함

**기준:** 학습자가 흔히 하는 실수와 그 해결 방법을 포함하면 학습 효과가 크게 올라간다.

**예:**
```
❌ 흔한 실수: `model.train()`을 빼먹고 평가 모드에서 학습
✅ 정상: `model.train()` 호출 후 `dropout` 활성화
📌 이유: eval 모드에서는 dropout이 비활성화되어 학습 불안정
```

**검토 시 확인할 점:**
- 학습자가 자주 하는 실수가 1개 이상 포함되었는가
- 오답이 단순히 틀렸다고 표시하는 대신, 왜 틀렸고 어떻게 고치는지 설명하는가

## 4. 구조와 흐름

수업자료는 자습서처럼 스스로 따라갈 수 있어야 한다.

### 4.1 고정 구조

**기준:** 수업자료는 아래 5개 섹션을 반드시 포함해야 한다.

| 섹션 | 설명 | 비중 |
|------|------|------|
| **학습 목표** | 무엇을 할 수 있게 되는가 | 5% |
| **개념 설명** | 핵심 개념 이해 | 25% |
| **예제** | 개념을 적용한 구체적 사례 | 30% |
| **실습** | 직접 해보는 과제 | 30% |
| **요약** | 핵심 포인트 정리, 다음 학습 방향 | 10% |

**검토 시 확인할 점:**
- 5개 섹션이 모두 존재하는가
- 개념 설명 밖에 없고 실습이 없는 수업자료는 완성되지 않은 것
- 예제와 실습이 전체의 60% 이상을 차지하는가

### 4.2 진행 가이드

**기준:** 독자가 자료를 어떻게 사용해야 하는지 안내해야 한다.

**예:**
- "이 자료를 보려면 Python 3.12 이상이 설치되어 있어야 합니다"
- "각 예제를 직접 실행해보면서 따라오세요 (총 소요 시간: 약 30분)"
- "실습 과제는 정답이 여러 개니 자신의 방식대로 풀어보세요"

**검토 시 확인할 점:**
- 수업 시작 전 준비 사항이 명시되었는가
- 예상 소요 시간이 표시되었는가
- 난이도 표시(L1-L4)가 있는가

### 4.3 자기 완결성

**기준:** 수업자료는 외부 자료에 의존하지 않고, 이 자료 하나만으로 학습이 가능해야 한다.

**검토 시 확인할 점:**
- "자세한 내용은 공식 문서를 참고하세요"로 떠넘기는 구간이 있는가
- 필요한 지식은 수업자료 내에서 모두 설명되었는가 (선수 지식 범위에서)
- 링크는 추가 정보용이지, 핵심 내용을 대체하지 않는가

## 5. 금지 규칙

### 5.1 금지 표현

| 금지 | 이유 | 대체 |
|------|------|------|
| "참고로" | 딴길로 새는 느낌 | 그냥 설명에 포함 |
| "여기서 중요한 점은" | 문맥이 중요함을 드러내야 함 | 별도 단락/강조로 처리 |
| "쉽게 말하면" | 처음부터 쉽게 쓰면 됨 | 바로 쉬운 설명 |
| "이해하기 쉽게" | 말로 설명하지 말고 구조로 보여줘라 | 수준에 맞춰 작성 |
| "~~에 대해 알아보자" | 클리셰 | "~~를 이해하려면 ~~부터 알아야 한다" |
| "그냥/대충/아마" | 부정확한 표현, 학습자 혼란 | 정확히 서술 |
| "매우/정말/너무" | 강조 불필요 | 구체적인 수치나 근거로 |

### 5.2 주관적 감상 금지

**기준:** 수업자료에는 작성자의 개인적 견해나 감상이 포함되지 않는다.

| 금지 | 이유 |
|------|------|
| "재미있다" | 주관적 감상, 학습과 무관 |
| "유용하다" | 배우는 사람이 판단할 일 |
| "실무에 도움이 된다" | 상황에 따라 다름 |
| "이 방법이 더 좋다" | 이유와 근거 없이 좋다고만 하면 안내가 아니라 의견 |

### 5.3 설명 부족 금지

**기준:** 아래 패턴은 학습자의 이해를 방해하므로 금지.

| 금지 패턴 | 예시 | 문제점 |
|----------|------|--------|
| "생략" | "나머지는 비슷하니 생략한다" | 학습자 혼란 |
| "쉽게 알 수 있다" | "코드를 보면 쉽게 알 수 있다" | 천재 가정 |
| "당연히" | "당연히 이렇게 동작한다" | 초심자에게 당연하지 않음 |
| "자명하다" | "수식을 보면 자명하다" | 입문자 배제 |

### 5.4 서술 금지

| 금지 | 이유 |
|------|------|
| 한자 | 전면 금지 |
| ASCII 그림/표 | 텍스트 기반 시각 자료 금지 |
| 수식 설명 없는 코드 덩어리 | 코드는 예제일 뿐, 설명이 함께 있어야 함 |
| 확인 장치 없는 학습 목표 | 목표 설정만 하고 확인하지 않으면 무의미 |

## 부록: 검토 결과 판정 기준

각 기준 항목에 대해 아래 3단계로 판정한다.

| 판정 | 의미 | 조치 |
|------|------|------|
| **통과** | 기준 충족 | 수정 불필요 |
| **경고** | 기준에 미달하나 수업자료로서 가치는 유지 | 수정 권장, 생략 가능 |
| **실패** | 기준 미달로 수업자료의 교육 목적을 달성하지 못함 | 반드시 수정 후 재검토 |

**최종 판정 기준:**
- 실패가 0개이고 경고가 3개 이하 → **통과**
- 실패가 1개 이상 → **수정 필요** (모든 실패 항목 해결 후 재검토)
- 경고가 4개 이상 → **개선 권장**

### 검토 보고서 양식

