# TIL 문서 품질 기준

> 이 문서는 TIL(Today I Learned) 포스팅이 갖춰야 할 최소 품질 기준을 정의합니다.
> 모델이 중간 생성 결과물을 검토할 때, 아래 각 항목을 기준으로 현재까지 작성된 내용이 조건에 부합하는지 판단하는 용도로 사용됩니다.
> 각 기준은 "왜 중요한가" → "구체적 기준" → "위반 시 문제점" 순서로 설명합니다.

---

## 목적
오늘 배운 내용을 간결하게 기록. 깊이 있는 분석이나 회고가 아닌 기록에 충실.

## 0. 제목 헤딩

**기준:** 문서의 가장 위에는 세션 제목을 `#` H1 헤딩으로 지정해야 한다.

**검토 시 확인할 점:**
- 문서 첫 줄이 `# {세션 제목}` 형식의 H1 헤딩인가
- 본문 내 다른 헤딩이 H1(`#`)을 중복 사용하지 않는가 (본문은 `##` 이하 사용)

## 1. 기록의 충실성

TIL의 핵심은 오늘 배운 것을 정확히 기록하는 것이다. 분석이나 회고가 아니라, 배운 사실을 그대로 옮기는 데 집중한다.

### 1.1 사실 중심

**기준:** 모든 내용은 검증 가능한 사실이어야 한다. 추측, 의견, 감상은 TIL에 포함되지 않는다.

| 나쁨 | 좋음 |
|------|------|
| "파이썬 3.12가 좋아진 것 같다" | "Python 3.12에서 PEP 695 타입 파라미터 문법이 안정화되었다" |
| "async/await가 유용해 보인다" | "asyncio.gather()로 3개의 API 호출을 병렬 처리: 총 시간 1.2초 → 0.4초" |
| "리팩토링이 중요하다는 걸 다시 느꼈다" | "함수 분리 후 테스트 커버리지가 62%에서 89%로 증가" |

**검토 시 확인할 점:**
- "~인 것 같다", "~처럼 보인다" 같은 불확실한 표현이 있는가
- 주장을 뒷받침하는 코드/수치/출처가 있는가
- 개인적 의견과 사실이 혼용되지 않았는가

### 1.2 배운 것만 적을 것

**기준:** TIL은 "오늘 배운 것"에 한정된다. 이미 알고 있던 내용을 재구성한 것이 아니라, 오늘 새롭게 알게 된 사실이어야 한다.

**위반 사례:**
- 이미 1년간 써온 기술을 TIL 형식으로 정리
- 다른 사람의 경험/지식을 배운 척 옮겨씀
- 책 내용 요약 — "오늘 ~라는 것을 배웠다" 프레임만 있고 실제로 배운 게 없음

**검토 시 확인할 점:**
- 이 내용을 작성자가 오늘 처음 알게 되었는가 (문맥상 타당한가)
- 기존에 알던 내용을 단순히 글로 다시 쓰고 있지 않은가

## 2. 간결성

TIL은 깊이보다 폭과 속도가 중요하다. 에세이처럼 풀어쓰거나 딥다이브처럼 깊게 파지 않는다.

### 2.1 길이 제한

**기준:** TIL 한 편의 적정 길이는 아래 범위를 초과하지 않아야 한다.

| 항목 | 제한 |
|------|------|
| 전체 길이 | A4 반 페이지 이내 (약 200-500단어) |
| 항목 수 | 1-3개 (하루에 배운 모든 것을 다 담으려 하지 않음) |
| 코드 블록 | 있어도 10-20줄 이내 |

**검토 시 확인할 점:**
- 글이 너무 긴가 (A4 1페이지 이상 → 일부는 다음 TIL로 분리하거나 딥다이브로 승격)
- 하나의 TIL에 너무 많은 주제를 담고 있지 않은가 (3개 초과 → 분리)

### 2.2 바로 본론

**기준:** 도입부, 배경 설명, 역사적 맥락 없이 핵심 내용으로 바로 진입해야 한다.

| 나쁨 | 좋음 |
|------|------|
| "오늘은 파이썬의 typing 모듈에 대해 알아보았다" | "Python 3.12: `type Vector[T] = list[T]` — 타입 별칭 문법 간소화" |
| "정규표현식은 문자열 패턴 매칭에 사용되며..." | "정규식에서 `(?P<name>...)` 명명 그룹: `m.group('name')`으로 접근" |

**검토 시 확인할 점:**
- 첫 문장이 핵심 내용인가 (배경 설명으로 시작하지 않는가)
- "~에 대해 알아보았다" 같은 군더더기가 있는가

### 2.3 블릿 우선

**기준:** TIL은 서술형 문단보다 블릿 포인트나 짧은 코드 예제를 선호한다.

| 나쁨 | 좋음 |
|------|------|
| 긴 문단으로 설명 | `- 핵심 개념: 한 줄 요약 → 코드 예제` |

**검토 시 확인할 점:**
- 3문장 이상의 문단이 TIL에 존재하는가 (있다면 딥다이브/에세이가 아닌지 의심)
- 각 블릿이 하나의 완결된 정보 단위인가
- 코드 예제가 있다면 설명과 1:1로 대응하는가

## 3. 정확성

TIL은 짧지만 정확해야 한다. 부정확한 TIL은 시간 낭비를 넘어 잘못된 지식을 심어준다.

### 3.1 코드 검증

**기준:** TIL에 포함된 모든 코드 예제는 실제로 실행되어 검증된 것이어야 한다.

**검토 시 확인할 점:**
- 코드에 문법 오류가 없는가
- import 누락이 없는가
- 코드 실행 결과가 설명과 일치하는가

### 3.2 버전 명시

**기준:** 특정 버전에 종속된 내용이라면 반드시 버전을 명시해야 한다.

**예:** "Python 3.12+에서만 동작", "PyTorch 2.0 기준", "macOS Sequoia 15.4에서 확인"

**검토 시 확인할 점:**
- 버전에 따라 동작이 달라지는 내용인가 → 버전 명시
- 미래에 깨질 수 있는 지식인가 (API deprecated 등) → 버전과 함께 경고 표기

### 3.3 인용

**기준:** 다른 사람의 지식/코드/아이디어를 참고했다면 반드시 출처를 표시해야 한다.

**검토 시 확인할 점:**
- 공식 문서/스택오버플로우/블로그에서 가져온 내용에 링크가 있는가
- "~에서 발견"이라는 단서 없이 타인의 코드가 그대로 쓰이지 않았는가

## 4. 구조화

TIL은 단순하지만 정보를 쉽게 찾을 수 있도록 구조화되어야 한다.

### 4.1 태그/키워드

**기준:** TIL의 첫 줄 또는 상단에 분류 태그가 있어야 한다.

**예:**
```
[[Python]] 3.12 PEP 695 타입 파라미터 문법 안정화
[[Docker]] --platform linux/amd64 옵션으로 크로스 플랫폼 빌드
[[Git]] git bisect로 최초 버그 발생 커밋 찾기
```

**검토 시 확인할 점:**
- 태그가 일관된 형식인가
- 태그가 너무 세분화(기술명-하위기능)되어 있지 않은가

### 4.2 카테고리화

**기준:** TIL의 내용은 아래 중 하나의 성격을 가져야 하며, 그에 맞게 정리되어야 한다.

| 성격 | 설명 | 예시 |
|------|------|------|
| 새 기술 습득 | 처음 써본 도구/라이브러리 | "pytest fixture scope: session > module > function" |
| 문제 해결 | 오늘 겪은 버그와 해결 | "macOS에서 .env 파일이 안 읽히는 문제 → dotenv 경로 절대 경로로" |
| 발견/팁 | 우연히 알게 된 효율적 방법 | "VS Code Ctrl+D 연속 선택으로 일괄 수정" |
| 개념 이해 | 오늘 이해한 개념 정리 | "Python descriptor protocol: `__get__`, `__set__`이 호출되는 시점" |

**검토 시 확인할 점:**
- 내용의 성격이 명확한가
- 성격에 맞지 않게 장황하게 쓰이지 않았는가 (예: 문제 해결인데 개념 설명이 80%)

## 5. 금지 규칙

### 5.1 금지 표현

**기준:** TIL에서 가치를 더하지 않는 표현은 금지.

| 금지 | 이유 |
|------|------|
| "오늘은 ~에 대해 알아보았다" | 불필요한 도입, 생략 |
| "재미있었다/유용했다" | 개인적 감상, TIL 불필요 |
| "다음과 같이 정리할 수 있다" | 그냥 정리하면 됨 |
| "참고로" | 출처면 출처, 본문이면 본문 |
| "더 자세히 알고 싶다면" | TIL이 요약이니까 당연함 |
| "~다고 생각한다" (의견 표현) | 사실만 적어라 |
| "매우/정말/너무/진짜" | 강조 불필요 |

### 5.2 포함 금지

**기준:** TIL의 성격을 벗어나는 내용은 포함하지 않는다.

| 금지 | 이유 |
|------|------|
| 개인적 감상/회고 | TIL이 아니라 일기 |
| 배경 설명 3문장 이상 | 딥다이브로 가야 함 |
| 역사/비교 분석 | 필요하면 링크로 대체 |
| 프로젝트 진행 보고 | TIL이 아니라 업무 일지 |
| 한자 | 전면 금지 |
| ASCII 그림/표 | 텍스트 기반 시각자료 금지 |

### 5.3 중복 금지

**기준:** 이전 TIL과 동일한 내용을 반복하지 않는다. 단, 더 깊이 배웠거나 새로운 관점이 있다면 "이전 TIL 보완" 표시 후 작성 가능.

**검토 시 확인할 점:**
- 같은 주제로 이전에 TIL을 작성한 적이 있는가
- 새로운 내용이 이전 TIL의 70% 이상 중복되는가 → 작성 중단 또는 "업데이트" 표기

## 부록: 검토 결과 판정 기준

각 기준 항목에 대해 아래 3단계로 판정한다.

| 판정 | 의미 | 조치 |
|------|------|------|
| **통과** | 기준 충족 | 수정 불필요 |
| **경고** | 기준에 미달하나 TIL로서 가치는 유지 | 수정 권장, 생략 가능 |
| **실패** | 기준 미달로 TIL의 목적(기록/전달)을 달성하지 못함 | 반드시 수정 후 재검토 |

**최종 판정 기준:**
- 실패가 0개이고 경고가 2개 이하 → **통과**
- 실패가 1개 이상 → **수정 필요**
- 경고가 3개 이상 → **개선 권장** (TIL의 간결성 원칙 훼손)

### 검토 보고서 양식

```
## 검토 결과: {통과/수정 필요/개선 권장}

### 실패 항목
1. [기준명] — {구체적 위치} — {문제점}

### 경고 항목
1. [기준명] — {구체적 위치} — {문제점}

### 종합 의견
- 강점:
- 개선 필요:
- 권장 조치:
```
