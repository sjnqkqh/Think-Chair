# 에세이 문서 품질 기준

> 이 문서는 기술 블로그 에세이 포스팅이 갖춰야 할 최소 품질 기준을 정의합니다.
> 모델이 중간 생성 결과물을 검토할 때, 아래 각 항목을 기준으로 현재까지 작성된 내용이 조건에 부합하는지 판단하는 용도로 사용됩니다.
> 각 기준은 "왜 중요한가" → "구체적 기준" → "위반 시 문제점" 순서로 설명합니다.

---

## 목적
개인적 경험과 통찰을 진솔하게 공유하는 글. 기술적 사실을 바탕으로 개발자로서의 경험담을 풀어내는 것이 목적. 일반론이나 교훈 나열이 아니라 '나만의 이야기'여야 함.

## 0. 제목 헤딩

**기준:** 문서의 가장 위에는 세션 제목을 `#` H1 헤딩으로 지정해야 한다.

**검토 시 확인할 점:**
- 문서 첫 줄이 `# {세션 제목}` 형식의 H1 헤딩인가
- 본문 내 다른 헤딩이 H1(`#`)을 중복 사용하지 않는가 (본문은 `##` 이하 사용)

## 1. 진정성

에세이의 핵심 가치는 진정성이다. 기술 문서처럼 정확함보다, 개인의 경험과 감정이 얼마나 진실되게 전달되는가가 중요하다.

### 1.1 개인 경험 기반

**기준:** 모든 내용은 작성자의 실제 경험에서 출발해야 한다. 일반론이나 교훈 나열은 에세이가 아니다.

| 나쁨 | 좋음 |
|------|------|
| "개발자에게 테스트 코드는 중요하다" | "내가 테스트 코드 없이 프로젝트를 진행했다가 장애를 겪은 일" |
| "좋은 팀워크의 중요성" | "내 PR이 3일째 리뷰되지 않아서 직접 찾아가 이야기했던 경험" |
| "배움에는 끝이 없다" | "10년 차가 되니 오히려 모르는 게 더 많아진 느낌" |

**검토 시 확인할 점:**
- 글의 주장/깨달음이 구체적인 개인 경험에서 비롯되었는가
- "~는 중요하다" 같은 격언/일반론으로 대체하고 있지 않은가
- "우리는"이 아니라 "나는"으로 쓰였는가

### 1.2 감정의 구체성

**기준:** 감정 표현은 "좋았다/나빴다/힘들었다" 같은 추상적 형용사가 아니라, 구체적인 신체 반응이나 상황 묘사로 대체되어야 한다.

| 나쁨 | 좋음 |
|------|------|
| "좋았다" | "처음 내 코드가 프로덕션에서 동작하는 모습을 보며 손이 떨렸다" |
| "힘들었다" | "새벽 3시, 에러 로그 200줄 앞에서 커피를 한 잔 더 따랐다" |
| "재미있었다" | "문제를 풀고 나니 시간이 3시간이나 지나있었다" |

**검토 시 확인할 점:**
- 감정이 형용사 하나로 처리되지 않았는가
- 감정의 원인이 되는 구체적인 상황이 묘사되었는가
- 독자가 공감할 수 있는 디테일이 있는가

### 1.3 과장과 극적 연출 금지

**기준:** 실제 경험을 미화하거나 극적으로 각색하면 진정성이 사라진다.

| 나쁨 | 좋음 |
|------|------|
| "팀원 모두가 감동했다" | "두 명이 '이렇게 하면 되겠네'라고 말했다" |
| "완벽한 해결책이었다" | "일단 동작은 했지만 구조가 깔끔하진 않았다" |
| "인생이 바뀌었다" | "이 경험 이후로 코드를 짜는 습관이 조금 바뀌었다" |

**검토 시 확인할 점:**
- 과장된 표현(완벽, 모두, 전혀, 인생역전)이 있는가
- 실패/실수/부족함을 솔직하게 드러내고 있는가
- 너무 깔끔하게 포장된 이야기는 아닌가

## 2. 구성

에세이는 이야기로서의 흐름이 있어야 한다. 기술 보고서 같은 단조로운 구조는 에세이의 매력을 떨어뜨린다.

### 2.1 에세이 구조 (도입-전개-깨달음-마무리)

**기준:** 에세이는 아래 4개 단계를 자연스럽게 따라야 한다. 각 섹션은 엄격히 분리되기보다 유기적으로 연결되어야 한다.

| 단계 | 역할 | 비중 |
|------|------|------|
| **도입** | 독자를 이야기 속으로 끌어들임. 상황/맥락/의문 제시 | 10-15% |
| **전개** | 경험을 시간 순서나 논리 순서로 풀어냄. 구체적 장면과 대화 | 50-60% |
| **깨달음** | 경험을 통해 얻은 인사이트. 일반론이 아닌 개인적 깨달음 | 15-20% |
| **마무리** | 여운을 남기는 마무리. 독자에게 질문이나 생각할 거리 제시 | 10-15% |

**검토 시 확인할 점:**
- 4개 단계가 모두 존재하는가
- 전개 단계가 전체의 절반 이상을 차지하며 경험 자체에 집중하는가
- 깨달음이 일반론(개발자에게 중요한 것은...)이 아닌 개인적 통찰인가

### 2.2 흐름의 자연스러움

**기준:** 각 단락은 논리적으로 연결되어야 한다. 갑작스러운 전환이나 이야기의 단절이 없어야 한다.

**위반 사례:**
- 도입 없이 갑자기 깨달음부터 나오는 경우
- 경험을 충분히 풀지 않고 바로 교훈으로 넘어가는 경우
- 시간 순서가 뒤섞여 독자가 맥락을 잃는 경우

**검토 시 확인할 점:**
- "그런데", "그래서", "그러던 중" 같은 연결 장치가 자연스러운가
- 각 문단이 이전 문단의 내용을 받아서 발전시키는가
- 순서를 바꿔도 말이 되는가 (바꿔도 말이 되면 구조에 문제가 있는 것)

## 3. 문장 품질

에세이는 기술 문서 이상으로 문장의 맛과 리듬이 중요하다.

### 3.1 문장의 다양성

**기준:** 문장의 길이와 구조가 다양해야 단조롭지 않다.

| 문제 | 예시 |
|------|------|
| 같은 길이 반복 | "그때 나는 ~였다. 그리고 ~했다. 그래서 ~되었다." |
| 같은 접속사 반복 | "그런데 ... 그런데 ... 그런데 ..." |
| 주어-서술어 패턴 고정 | "나는 ~했다. 나는 ~생각했다. 나는 ~결정했다." |

**검토 시 확인할 점:**
- 3문장 이상 같은 패턴(길이/시작/구조)이 반복되는 구간이 있는가
- 간결한 문장과 약간 긴 문장이 교차하는가
- 같은 접속사가 단락 내에서 2번 이상 등장하는가

### 3.2 감각적 묘사

**기준:** 시각, 청각, 촉각 등 감각을 자극하는 표현이 포함되어야 한다.

| 나쁨 | 좋음 |
|------|------|
| "야근을 했다" | "새벽 2시, 모니터 불빛만이 사무실을 비추고 있었다" |
| "팀원과 이야기했다" | "슬랙보다는 직접 가서 말하는 게 낫겠다 싶어 자리에서 일어났다" |
| "코드를 고쳤다" | "백스페이스를 누르는 손가락이 망설여졌다. 이게 맞는 걸까" |

**검토 시 확인할 점:**
- 장면이 독자의 머릿속에 그려지는가
- 장소, 시간, 날씨, 분위기 중 하나라도 묘사가 있는가
- 추상적 묘사(예쁘다, 조용하다) 대신 구체적 묘사(노란 조명, 키보드 소리만)인가

### 3.3 문장 길이 제한

**기준:** 에세이는 숨 쉴 틈을 주는 문장 리듬이 중요하다. 너무 긴 문장은 몰입을 깬다.

| 항목 | 제한 |
|------|------|
| 한 문장 | 40단어 이내 |
| 한 문단 | 3-5문장 |
| 내포 깊이 | 2중 이하 |

**검토 시 확인할 점:**
- 40단어를 초과하는 문장이 있는가 → 분할
- 쉼표가 4개 이상인 문장이 있는가 → 분할
- 5문장 이상의 문단이 집중력을 유지하는가

## 4. 기술적 정확성

에세이는 개인적이지만, 언급하는 기술적 사실은 반드시 정확해야 한다.

### 4.1 기술 팩트 체크

**기준:** 에세이에서 언급된 모든 기술적 주장(라이브러리 버전, 측정값, 동작 원리)은 사실이어야 한다.

| 나쁨 | 좋음 |
|------|------|
| "파이썬은 GIL 때문에 멀티스레드가 안 된다" | "파이썬 GIL 때문에 CPU 바운드 작업에서는 멀티스레드 효과가 제한적이었다" |
| "쿠버네티스가 답이다" | "쿠버네티스로 전환했더니 배포 시간이 15분에서 3분으로 줄었다" |

**검토 시 확인할 점:**
- "~라고 알고 있다"는 표현이 불확실한 기술 주장을 감추고 있지 않은가
- 팩트가 틀리면 에세이 전체의 신뢰도가 무너진다는 점을 인지

### 4.2 의견과 사실 구분

**기준:** 개인적 의견과 기술적 사실은 반드시 구분되어야 한다. 의견에는 "내 생각에는" 같은 단서를 붙이거나 문맥상 의견임이 명확해야 한다.

**검토 시 확인할 점:**
- "~다"로 끝나는 문장이 모두 검증된 사실인가
- 개인적 인상(번거롭다, 불편하다)이 절대적 사실처럼 쓰이지 않았는가

## 5. 금지 규칙

### 5.1 금지 표현

**기준:** 에세이의 가치를 떨어뜨리는 표현을 금지한다.

| 금지 | 이유 | 대체 |
|------|------|------|
| "참고로" | 딴길로 새는 느낌 | 본문에 녹이거나 과감히 생략 |
| "여기서 중요한 점은" | 설명하지 말고 보여줘라 | 문단 자체로 중요성을 드러낼 것 |
| "한마디로 말하면" | 한마디로 안 되는 걸 에세이라고 쓴다 | 그냥 쓰면 됨 |
| "~~에 대해 알아보자" | 블로그 특유의 클리셰 | "~~를 겪었다"로 직접 시작 |
| "오늘은 ~을 주제로" | 일기장 아님 | 상황부터 시작 |
| "~다고 생각한다" 과다 사용 | 3회 이상이면 리듬 깨짐 | 다양한 표현 사용 |
| "매우/정말/너무/진짜" | 강조 표현은 쓰면 쓸수록 약해짐 | 구체적 묘사로 대체 |

### 5.2 한자 사용 금지

**기준:** 한자(U+4E00–U+9FFF)는 전면 금지. 사용자 원본 인용문은 제외.

**검토 방법:** 정규식 `[\u4E00-\u9FFF]`로 전체 문서 스캔.

### 5.3 텍스트 기반 시각 자료 금지

**기준:** ASCII 아트, 텍스트 그림, 공백 다이어그램 금지. 마크다운 테이블만 허용.

### 5.4 건조한 보고체 금지

**기준:** 에세이는 감성과 리듬이 있는 글이다. 아래 같은 보고서 스타일은 금지.

| 보고체 (금지) | 에세이체 (권장) |
|----------------|-----------------|
| "본 프로젝트에서는..." | "이번 프로젝트에서 나는..." |
| "다음과 같은 이슈가 있었다" | "예상치 못한 문제가 터졌다" |
| "결론적으로 개선이 필요하다" | "아직 갈 길이 멀다는 생각이 들었다" |
| 수동태 문장 | 능동태 문장 (내가 주체) |

## 부록: 검토 결과 판정 기준

각 기준 항목에 대해 아래 3단계로 판정한다.

| 판정 | 의미 | 조치 |
|------|------|------|
| **통과** | 기준 충족 | 수정 불필요 |
| **경고** | 기준에 미달하나 내용 이해에 큰 지장 없음 | 수정 권장, 생략 가능 |
| **실패** | 기준 미달로 내용의 진정성이나 가독성이 크게 저하됨 | 반드시 수정 후 재검토 |

**최종 판정 기준:**
- 실패가 0개이고 경고가 3개 이하 → **통과**
- 실패가 1개 이상 → **수정 필요**
- 경고가 4개 이상 → **개선 권장**

### 검토 보고서 양식

```
## 검토 결과: {통과/수정 필요/개선 권장}

### 실패 항목
1. [기준명] — {구체적 위치} — {문제점}

### 경고 항목
1. [기준명] — {구체적 위치} — {문제점}

### 종합 의견
- 강점:
- 개선 필요:
- 권장 조치:
```
