# 기술 딥다이브 문서 품질 기준

> 이 문서는 기술 딥다이브 포스팅이 갖춰야 할 최소 품질 기준을 정의합니다.
> 모델이 중간 생성 결과물을 검토할 때, 아래 각 항목을 기준으로 현재까지 작성된 내용이 조건에 부합하는지 판단하는 용도로 사용됩니다.
> 각 기준은 "왜 중요한가" → "구체적 기준" → "위반 시 문제점" 순서로 설명합니다.

---

## 목적

하나의 기술/개념을 일상 지식보다 깊은 층위까지 파고드는 글. 단순 사용법이 아니라 핵심 원리와 트레이드오프까지 이해하는 것이 목적.

**딥다이브의 본질은 "깊이"이며, 그 깊이는 여러 형태의 근거로 확보된다.** 수식과 증명, 실행 가능한 구현 코드, 벤치마크 수치, 논문·실험 인용, 현실 예시, 개념의 정밀한 분해가 모두 근거가 된다. 주제에 따라 어떤 근거가 적합한지 달라질 뿐, 어느 하나가 필수인 것은 아니다.

따라서 핵심 판단 기준은 "코드나 벤치마크가 있는가"가 아니라 **"일상 지식보다 깊은 근거로 주장을 뒷받침하는가"**이다. 구현 코드가 없다는 이유만으로 미달 판정을 내려서는 안 된다. 반대로 코드가 있더라도 근거 없이 나열만 했다면 깊이가 부족한 것이다.

## 0. 제목 헤딩

**기준:** 문서의 가장 위에는 세션 제목을 `#` H1 헤딩으로 지정해야 한다.

**검토 시 확인할 점:**
- 문서 첫 줄이 `# {세션 제목}` 형식의 H1 헤딩인가
- 본문 내 다른 헤딩이 H1(`#`)을 중복 사용하지 않는가 (본문은 `##` 이하 사용)

## 1. 사용자 이해 수준

딥다이브는 일반 독자를 위한 설명문이 아니라, 사용자와의 대화를 통해 사용자의 이해를 더 깊게 만드는 과정이다. 문서의 깊이는 대화에서 드러난 사용자의 이해와 질문을 바탕으로 확보한다.

### 1.1 현재 이해 파악

**기준:** 대화에서 사용자가 현재 알고 있는 것, 확신하지 못하는 것, 설명할 수 없는 것을 구분해야 한다.

| 불충분한 파악 | 충분한 파악 |
|---------------|-------------|
| "사용자는 AI에 관심이 있다" | "사용자는 어텐션의 역할은 설명하지만, Q·K·V 연산과 softmax의 의미는 연결하지 못한다" |
| "사용자는 백엔드 개발자다" | "사용자는 FastAPI의 비동기 처리 경험을 근거로 이벤트 루프의 동작을 설명한다" |

**검토 시 확인할 점:**

- 사용자가 현재 이해하는 범위와 확인이 필요한 범위가 대화에서 드러났는가
- 직무나 경력 같은 일반적 분류 대신, 사용자의 실제 설명과 근거를 바탕으로 판단했는가

### 1.2 대화에서 드러난 빈틈

**기준:** 사용자가 이미 설명한 내용을 반복하기보다, 이해의 빈틈과 근거가 부족한 지점을 질문과 문서에서 다뤄야 한다.

**예:** "사용자는 행렬 곱의 계산은 설명했지만, 왜 어텐션 점수에 스케일링이 필요한지는 설명하지 못했다. 문서에서는 이 근거를 수식과 수치로 다룬다."

**검토 시 확인할 점:**

- 사용자가 이미 제시한 설명을 일반론으로 되풀이하고 있지 않은가
- 사용자가 설명하지 못했거나 근거가 부족했던 핵심을 문서에서 깊게 다뤘는가

## 2. 문서 구조

구조가 없는 딥다이브는 독자가 맥락을 잃고 중간에 포기하게 된다.

### 2.1 섹션 수 (8-10개)

**기준:** 하나의 딥다이브는 8~10개의 섹션으로 구성되어야 한다. 5개 미만이면 얕고, 15개 이상이면 산만하다.

| 섹션 수      | 판정                        |
|-----------|---------------------------|
| 4개 이하     | 너무 얕음 — 심층 분석이라고 보기 어려움   |
| 5-7개      | 다소 부족 — 핵심 섹션이 누락되었을 가능성  |
| **8-10개** | **적정**                    |
| 11-14개    | 약간 많음 — 통합 가능한 섹션이 있는지 확인 |
| 15개 이상    | 산만함 — 일부는 부록/참고로 분리해야 함   |

**검토 시 확인할 점:**

- 섹션 수가 적정 범위인가
- 각 섹션이 독립적인 가치가 있는가 (하나의 섹션이 한 가지 질문에 답하는가)
- 섹션을 더 통합하거나 분할할 필요는 없는가

### 2.2 섹션 순서

**기준:** 섹션은 "문제 정의 → 직관적 설명 → 상세 메커니즘 → 근거 → 한계 → 결론" 순서를 따라야 한다.

```
1. 문제 정의 — 이 기술/개념이 해결하는 것이 무엇인가
2. 직관적 이해 — 비유나 은유로 개념 잡기
3. 상세 메커니즘 — 수식, 아키텍처, 핵심 로직, 개념의 정밀한 분해
4. 근거 — 주장을 뒷받침하는 깊은 근거. 주제에 맞는 형태를 쓴다:
         실행 가능한 코드, 벤치마크 수치, 논문·실험 인용, 현실 예시 등
5. 한계 — 이 기술/개념이 적합하지 않은 경우, 그리고 개선 방향
6. 결론 — 핵심 주장을 요약하고 명확하게 마무리
```

**위반 사례:**

- 직관적 설명 없이 바로 수식으로 들어감 → 직관 섹션 누락 (초보자에게 치명적)
- 한계를 언급하지 않고 장점만 나열 → 한계 섹션 누락
- 결론 없이 한계에서 글이 끊김 → 결론 섹션 누락
- 주장을 뒷받침하는 근거가 전혀 없음(코드도, 수치도, 인용도, 현실 예시도 없음) → 근거 섹션 누락
- **주의:** 구현 코드가 없다는 사실 자체는 위반이 아니다. 코드든 인용이든, 주제에 맞는 근거로 깊이를 확보했는지를 본다.

**검토 시 확인할 점:**

- 6개 섹션(문제 정의·직관·상세 메커니즘·근거·한계·결론)이 모두 존재하는가
- 근거 섹션이 주제에 맞는 형태의 깊은 근거를 담고 있는가
- 순서가 논리적인가 (직관 없이 수식부터 시작하지 않는가)
- 독자가 순서대로 읽었을 때 이해도가 점진적으로 올라가는가

### 2.3 섹션별 설명 형식

**기준:** 각 섹션이 어떤 형식(코드/수식/비교표/다이어그램)으로 설명되어야 하는지가 정의되어야 한다.

| 섹션      | 권장 형식                     | 설명                          |
|---------|---------------------------|-----------------------------|
| 문제 정의   | 텍스트                       | 배경과 동기를 서술형으로               |
| 직관      | 비유/은유                     | 실제 경험에 빗대어 설명               |
| 상세 메커니즘 | 수식+코드+다이어그램 또는 정밀한 개념 분해 | 핵심은 수식/논리, 이해는 코드/예시        |
| 근거      | 주제에 맞는 형태(코드/수치/인용/현실 예시) | 개념 뒤에 즉시 예시, 수치 뒤에 즉시 해석    |
| 한계      | 텍스트+사례                    | 구체적인 실패 사례와 개선 방향 포함        |
| 결론      | 텍스트                       | 핵심 주장을 요약하고 명확히 마무리         |

**검토 시 확인할 점:**

- 각 섹션이 권장 형식에 맞게 작성되었는가
- 근거 섹션이 서술만 있고 실제 근거(코드/수치/인용/현실 예시)가 없지 않은가
- 성능/효율을 주장한다면 구체적 수치로 뒷받침되는가

## 3. 깊이 = 근거 밀도

딥다이브의 깊이는 추상적 등급이 아니라 **"주장에 근거가 얼마나 붙어 있는가"**로 측정한다. 근거 없이 단정만 나열하면 아무리 길어도 얕은 글이고, 핵심 주장마다 근거가 붙으면 깊은 글이다.

### 3.1 근거의 정의

**기준:** 다음 중 하나면 "근거 1개"로 센다. 형태는 주제에 맞게 고르며, 어느 하나가 우월한 것은 아니다.

- 수식 유도 또는 증명 스케치
- 실행 가능한 코드 또는 재현 가능한 실험
- 정량 수치 (벤치마크, 측정값) — 측정 환경과 함께
- 논문·표준 문서 인용 — 출처를 특정할 수 있게
- 구체적인 현실 예시 — 독자가 실제로 마주할 법한 시나리오

단순 재진술("중요하다", "효율적이다"), 개인 감상, 출처 없는 일반론은 근거로 세지 않는다.

### 3.2 근거 밀도 기준

**기준:** 주장의 중요도에 따라 아래 밀도를 충족해야 한다. 근거는 문단 단위로 카운트해서 판정한다.

- **핵심 주장** (이 문서가 증명하려는 것): 서로 다른 근거 2개 이상
- **일반 주장** (핵심을 뒷받침하는 것): 근거 1개 이상
- **근거 없는 단정의 비율**: 전체 주장 문단의 20% 이하

**검토 시 확인할 점:**

- 각 문단의 주장에 §3.1의 근거가 붙어 있는가 → 붙은 문단 수를 센다
- 핵심 주장에 근거가 2개 이상 붙어 있는가 (1개뿐이면 보완)
- 근거 없는 단정이 전체의 20%를 넘는가 → 초과 시 미달
- 근거가 핵심 주장이 아니라 부차적 내용(역사/배경)에 몰려 있지 않은가

## 4. 코드 품질

코드 예제가 포함된 딥다이브에서 코드가 실제로 동작하지 않으면 글 전체의 신뢰도가 무너진다.

> **적용 범위:** 이 섹션은 **코드가 포함된 경우에만** 적용한다. 코드 없이 근거(수식·인용·현실 예시)로 깊이를 확보한 문서는 이 섹션의 미달 항목으로 판정하지 않는다. "코드가 없다"는 사실 자체는 위반이 아니다.

### 4.1 실행 가능성

**기준:** 모든 코드 블록은 import부터 시작해서 복붙 후 에러 없이 실행 가능해야 한다.

**검토 시 확인할 점:**

- 첫 줄이 `import ...`로 시작해서 마지막 줄이 `print(...)` 또는 함수 호출로 끝나는가
- 필요한 의존성(pip install)이 명시되었는가
- 누락된 import가 없는가
- 정의된 함수는 반드시 호출 예제까지 있는가 (정의만 있고 호출 없음 = 미완성)

### 4.2 실제 데이터 사용

**기준:** 무작위 데이터나 MNIST 같은 장난감 데이터셋만 사용하지 말고, 실제 프로젝트에서 마주칠 법한 데이터를 사용해야 한다.

| 나쁨                                       | 좋음                                |
|------------------------------------------|-----------------------------------|
| `torch.randn(10, 512)`                   | "실제 뉴스 기사 1000건을 토크나이징한 결과"       |
| `sklearn.datasets.make_classification()` | "실제 매출 데이터 2024년 1월~12월 (CSV 로드)" |
| MNIST                                    | "자사 서비스 로그 데이터 10만 건"             |

**검토 시 확인할 점:**

- 데이터 생성 또는 로드 코드가 `실제 시나리오`를 반영하는가
- 테스트용 무작위 데이터를 사용한다면 그 이유가 명시되었는가

### 4.3 에러 처리

**기준:** 주요 예외 상황(try/except)과 edge case(빈 입력, None, 비정상 입력)에 대한 처리가 포함되어야 한다.

**검토 시 확인할 점:**

- 파일 입출력이 있는 경우 FileNotFoundError 처리
- API 호출이 있는 경우 timeout/connection error 처리
- 빈 리스트, None 입력에 대한 guard clause
- 배치 크기=1이나 시퀀스 길이=0 같은 edge case

### 4.4 코드 문서화

**기준:** 함수/클래스에 docstring(영문), 주요 변수에 shape 주석이 포함되어야 한다.

**예:**

```python
def forward(self, x: torch.Tensor) -> torch.Tensor:
    """Apply multi-head self-attention.
    
    Args:
        x: (batch, seq_len, d_model) input tensor
    Returns:
        (batch, seq_len, d_model) output tensor
    """
```

**검토 시 확인할 점:**

- 각 함수에 docstring이 있는가
- 주요 텐서 변수에 shape가 주석으로 표시되었는가
- 이해하기 어려운 로직에 인라인 주석이 있는가

## 5. 데이터와 수치

주장을 수치로 뒷받침하지 않는 딥다이브는 공학이 아니라 소설이다.

### 5.1 성능 주장의 수치화

**기준:** "성능이 좋다", "빠르다", "효율적이다" 같은 표현은 금지. 모든 성능/효율성 주장은 구체적인 수치로 제시되어야 한다.

| 나쁨                          | 좋음                                                   |
|-----------------------------|------------------------------------------------------|
| "Transformer가 RNN보다 성능이 좋다" | "WMT14 En-De에서 BLEU 41.8로 GNMT 대비 +17.3"             |
| "FP16 추론이 빠르다"              | "FP16 추론이 FP32 대비 1.8배 빠름 (A100, batch=32 기준)"       |
| "XGBoost가 성능이 좋다"           | "XGBoost AUC 0.873으로 Random Forest(0.821) 대비 +0.052" |

**검토 시 확인할 점:**

- "좋다/빠르다/효율적이다" 같은 표현이 수치 없이 사용되었는가
- 수치가 있다면 출처가 명시되었는가
- 측정 환경이 명시되었는가 (GPU 모델, 배치 크기, 데이터셋)

### 5.2 인용 형식과 수치 해석

**기준:** 외부 근거(논문·벤치마크)를 인용할 때는 출처를 특정할 수 있어야 하고, 수치를 제시한 뒤에는 그 수치가 무엇을 의미하는지 해석이 따라와야 한다.

- **인용 형식:** 논문은 저자, 제목, 식별자(arXiv ID/DOI 등), 연도를 기재한다. 수치 인용 시 조건을 명확히 한다 ("백본을 GPT-5로 고정했을 때 71.75% → 81.36%").
- **수치 뒤 해석:** 수치를 나열만 하지 않는다. 바로 뒤에 "이는 ~을 의미한다"는 해석을 붙인다. 예: "정답률이 71.75%에서 81.36%로 상승했다. 이는 성능 향상이 모델이 아닌 아키텍처에서 비롯됨을 보여준다."

**검토 시 확인할 점:**

- 인용된 논문의 출처를 독자가 특정할 수 있는가 (저자/식별자/연도)
- 수치가 조건 없이 제시되어 재현·비교가 불가능하지 않은가
- 수치 뒤에 해석 없이 다음 문단으로 넘어가고 있지 않은가

### 5.3 실험 환경 명시

**기준:** 모든 수치적 주장에는 측정 환경이 함께 명시되어야 한다.

**필수 포함 정보:**

- 하드웨어 (GPU: A100 80GB × 1, CPU: AMD EPYC 7B12)
- 소프트웨어 (PyTorch 2.1.0, CUDA 12.1)
- 데이터셋 (크기, 출처, train/val/test 분할)
- 하이퍼파라미터 (learning rate, batch size, epoch)

**검토 시 확인할 점:**

- 비교 대상 간 환경이 동일한가 (동일 GPU, 동일 데이터)
- 환경 정보가 누락된 수치가 있는가

### 5.4 정량 평가 기준

**기준:** 평가 기준이 모호하지 않고, 해당 도메인에서 표준적으로 사용되는 메트릭을 사용해야 한다.

| 도메인 | 표준 메트릭                                  |
|-----|-----------------------------------------|
| 분류  | Accuracy, Precision/Recall, F1, AUC-ROC |
| 생성  | BLEU, ROUGE, Perplexity                 |
| 검색  | Recall@k, MRR, NDCG                     |
| 회귀  | MSE, MAE, R²                            |
| 속도  | Latency (ms), Throughput (req/s)        |

**검토 시 확인할 점:**

- 평가 메트릭이 해당 도메인에 적합한가
- 하나의 메트릭에만 의존하지 않고 최소 2개 이상의 메트릭을 사용하는가

## 6. 제약 규칙

딥다이브의 가독성과 전문성을 해치는 요소를 방지한다.

### 6.1 금지 표현

**기준:** 딥다이브에서 가치를 더하지 않는 표현은 금지한다.

**금지 목록:**

- "참고로" — 불필요한 단서, 그냥 본문에 통합
- "여기서 중요한 점은" — 강조가 필요하면 별도 문단으로 분리
- "한마디로" — 딥다이브는 한마디로 설명할 수 없는 내용을 파고드는 것
- "이해하기 쉽게 말하자면" — 처음부터 쉽게 쓰면 됨
- "~~에 대해 알아보자" — "~~를 이해하려면 ~~부터 알아야 한다"로 대체

**검토 시 확인할 점:**

- 위 표현이 등장하는가 (발견 즉시 삭제)
- 문장에 불필요한 군더더기(쉼표 과다, 중복 수식어)가 없는가

### 6.2 한자 사용 금지

**기준:** 한자(U+4E00–U+9FFF)는 사용자의 가독성을 해치므로 전면 금지한다. 다만 사용자 원본 입력(인용 등)은 유지한다.

**예외:** 사용자가 작성한 인용문이나 코드 내 식별자.

**검토 방법:** 정규식 `[\u4E00-\u9FFF]`로 전체 문서를 스캔하여 검출.

**대체 표현 예시:**

| 한자 포함 | 올바른 표현              |
|-------|---------------------|
| 과다/과소 | 너무 많음/너무 적음         |
| 부족    | 모자람                 |
| 대비    | 비교                  |
| 이상/이하 | 보다 큼/보다 작음 또는 넘음/미만 |
| 상기    | 위에서 말한, 앞서          |

### 6.3 문단/문장 길이 제한

**기준:** 가독성을 위해 문단과 문장의 길이를 제한한다.

| 항목       | 제한               |
|----------|------------------|
| 한 문단     | 3-5문장            |
| 한 문장     | 50단어 이내          |
| 한 문장의 내포 | 2중 내포(접속사 2개) 이하 |

**검토 시 확인할 점:**

- 5문장을 초과하는 문단이 있는가 → 분할
- 쉼표로 길게 연결된 문장이 있는가 → 분할
- "~하고, ~하며, ~이고" 등이 3회 이상 반복되는 문장이 있는가 → 분할

### 6.4 텍스트 기반 시각 자료 금지

**기준:** 표(테이블)는 마크다운 형식도 포함해 금지. 텍스트로 그린 그림, ASCII 아트, 공백으로 만든 다이어그램도 금지.

- 비교/분류가 필요하면 글머리 기호 목록으로 서술 (금지: 마크다운 테이블 `| col1 | col2 |`, 텍스트로 그린 표 `+-----+-----+`)
- 이미지가 필요하면 이미지 URL(별도 업로드)로 대체 (금지: `[그림: 이 부분이 어텐션...]` 같은 텍스트 그림)
- 순서도가 필요하면 코드 블록이나 번호 목록으로 대체 (금지: 공백/별표로 만든 순서도)

## 부록: 검토 결과 판정 기준

각 기준 항목에 대해 아래 3단계로 판정한다.

| 판정     | 의미                           | 조치           |
|--------|------------------------------|--------------|
| **통과** | 기준 충족                        | 수정 불필요       |
| **경고** | 기준에 미달하나 내용 이해에 큰 지장 없음      | 수정 권장, 생략 가능 |
| **실패** | 기준 미달로 내용의 신뢰성이나 가독성이 크게 저하됨 | 반드시 수정 후 재검토 |

**최종 판정 기준:**

- 실패가 0개이고 경고가 3개 이하 → **통과**
- 실패가 1개 이상 → **수정 필요** (모든 실패 항목 해결 후 재검토)
- 경고가 4개 이상 → **개선 권장**

### 검토 보고서 양식

```
## 검토 결과: {통과/수정 필요/개선 권장}

### 실패 항목
1. [기준명] — {구체적 위치} — {문제점}

### 경고 항목
1. [기준명] — {구체적 위치} — {문제점}

### 종합 의견
- 강점:
- 개선 필요:
- 권장 조치:
```
