# 후기/회고 문서 품질 기준

> 이 문서는 후기(Review) 및 회고(Retrospective) 포스팅이 갖춰야 할 최소 품질 기준을 정의합니다.
> 모델이 중간 생성 결과물을 검토할 때, 아래 각 항목을 기준으로 현재까지 작성된 내용이 조건에 부합하는지 판단하는 용도로 사용됩니다.
> 각 기준은 "왜 중요한가" → "구체적 기준" → "위반 시 문제점" 순서로 설명합니다.

---

## 목적
특정 경험(프로젝트, 도구, 교육, 컨퍼런스 등)을 사실과 의견으로 정리하여, 같은 선택을 고민하는 사람에게 판단 기준을 제공. 광고가 아닌 객관적인 정보 전달이 목적.

## 0. 제목 헤딩

**기준:** 문서의 가장 위에는 세션 제목을 `#` H1 헤딩으로 지정해야 한다.

**검토 시 확인할 점:**
- 문서 첫 줄이 `# {세션 제목}` 형식의 H1 헤딩인가
- 본문 내 다른 헤딩이 H1(`#`)을 중복 사용하지 않는가 (본문은 `##` 이하 사용)

## 1. 사실과 의견의 분리

후기의 신뢰성은 사실과 의견을 얼마나 명확히 구분하는가에 달려 있다. 주관적 평가를 객관적 사실인양 포장하면 독자의 의사결정에 해를 끼친다.

### 1.1 사실 영역

**기준:** 측정 가능하거나 검증 가능한 모든 정보는 사실로 표기되어야 한다.

| 나쁨 | 좋음 |
|------|------|
| "응답 속도가 빨랐다" | "평균 응답 시간 120ms (p95 210ms)로 측정" |
| "비용이 저렴했다" | "월 $49로 유사 서비스(경쟁사 $79) 대비 38% 저렴" |
| "문서화가 잘 되어 있었다" | "공식 문서 200페이지, 예제 코드 50개 이상 제공" |

**검토 시 확인할 점:**
- 수치로 표현할 수 있는 내용을 형용사로 모호하게 처리했는가
- "빠르다/느리다/싸다/비싸다" 등 주관적 표현이 수치 없이 쓰였는가
- 비교 표현(낫다/좋다)에 비교 대상과 기준이 명시되었는가

### 1.2 의견 영역

**기준:** 의견/주관적 평가는 반드시 자신의 의견임을 드러내야 하며, 근거를 함께 제시해야 한다.

| 나쁨 | 좋음 |
|------|------|
| "UX가 별로다" | "UX가 아쉬웠다: 설정 화면까지 클릭 5회 필요, 경쟁사는 2회" |
| "추천한다" | "소규모 팀(5인 이하)이라면 추천한다: 설정 시간 10분으로 빠르고, 무료 티어로 충분히 테스트 가능" |
| "좋은 도구다" | "문서화와 커뮤니티가 활성화되어 있어 문제 해결이 쉬웠다" |

**검토 시 확인할 점:**
- 의견에 근거가 함께 있는가
- 의견이 적용 가능한 조건/범위(누가, 언제, 어떤 상황에서)와 함께 서술되었는가
- "좋다/싫다"로 끝나는 평가가 하나라도 있는가

### 1.3 극단적 표현 금지

**기준:** "최고", "최악", "완벽", "절대" 같은 극단적 표현은 사용하지 않는다.

| 나쁨 | 좋음 |
|------|------|
| "세상에서 가장 좋은 도구" | "내가 사용해본 도구 중에서는 가장 직관적이었다" |
| "절대 쓰지 마세요" | "OO한 상황에서는 비추천한다: 이유는..." |
| "완벽한 해결책" | "이 용도에는 적합하지만, 다른 용도에서는 제한이 있다" |

**검토 시 확인할 점:**
- 극단적 표현이 등장하는가
- 객관성을 유지하려는 노력이 글 전체에 일관되게 나타나는가

## 2. 구조

후기는 독자가 원하는 정보를 빠르게 찾을 수 있도록 구조화되어야 한다.

### 2.1 권장 구조

**기준:** 후기는 아래 5개 섹션을 따라야 한다. 각 섹션의 비중은 상황에 따라 조정 가능하나, 모든 섹션이 존재해야 한다.

| 섹션 | 포함해야 할 내용 |
|------|-----------------|
| **개요** | 무엇을, 언제, 왜, 어떤 조건에서 경험했는가 |
| **경험 상세** | 구체적으로 무엇을 했는가 — 설정/사용/도입 과정 사실적 서술 |
| **좋았던 점** | 수치/근거와 함께. 왜 좋았는지 (감정이 아니라 이유) |
| **아쉬웠던/실패한 점** | 마찬가지로 수치/근거와 함께. 문제와 원인을 함께 |
| **총평** | 종합 평가, 추천 대상과 조건, 대안 비교 |

**검토 시 확인할 점:**
- 5개 섹션이 모두 존재하는가
- 하나의 섹션이 지나치게 길거나 짧지 않은가 (예: 좋았던 점만 80%)
- 아쉬웠던 점이 없는가 → 있다면 신뢰성 하락

### 2.2 요약 정보

**기준:** 글의 상단에 요약 정보 박스(표)가 있어야 한다. 독자가 글 전체를 읽지 않고 핵심을 파악할 수 있어야 한다.

**예시:**
```
| 항목 | 내용 |
|------|------|
| 대상 | AWS Lambda + API Gateway |
| 기간 | 2025.09 - 2026.02 (6개월) |
| 인원 | 3명 (백엔드 2, 프론트 1) |
| 스택 | Python 3.12, FastAPI, DynamoDB |
| 결과 | API 응답 시간 320ms→145ms (54% 개선) |
```

**검토 시 확인할 점:**
- 요약 표가 상단에 있는가
- 요약 정보만 봐도 이 후기의 핵심을 알 수 있는가
- 결과에 구체적 수치(개선율, 기간, 규모)가 포함되었는가

## 3. 데이터와 근거

후기의 가치는 주장을 뒷받침하는 데이터의 질에 비례한다.

### 3.1 경험의 구체성

**기준:** "사용해봤다"는 어떤 의미인가? 구체적인 사용 조건과 기간이 명시되어야 한다.

| 나쁨 | 좋음 |
|------|------|
| "3개월 사용해봤다" | "3개월간 주 5일, 하루 4시간 이상 사용" |
| "프로젝트에 도입했다" | "MAU 5만 규모 서비스에 도입. 초기 설정 2일, 전체 전환 2주 소요" |
| "여러 기능을 써봤다" | "API 12개 중 10개를 실제 프로덕션에서 사용해봄" |

**검토 시 확인할 점:**
- 경험의 깊이(사용 시간, 범위, 기간)가 정량적으로 표현되었는가
- "써봤다/도입했다" 같은 모호한 표현에 머물지 않았는가

### 3.2 비교 기준

**기준:** 평가는 혼자 존재하지 않는다. 비교 대상(이전 도구, 경쟁사, 대안)이 명시되어야 한다.

**검토 시 확인할 점:**
- 이 도구/경험을 평가하기 위한 기준점(이전 상태, 경쟁 도구)이 있는가
- 비교가 "전/후" 또는 "이 도구/저 도구" 형태로 제시되었는가
- 비교 조건이 공정한가 (같은 규모, 같은 설정에서 비교했는가)

### 3.3 한계 인정

**기준:** 모든 기술/도구/경험에는 트레이드오프가 있다. 한계를 인정하지 않는 후기는 광고와 다르지 않다.

**검토 시 확인할 점:**
- 좋은 점 하나당 최소 하나의 아쉬운 점/한계가 있는가
- 특정 조건에서만 유효하다는 단서가 붙었는가 ("소규모에만 적합", "GPU가 있을 때만")
- 글의 전반적 톤이 "추천" 쪽으로만 치우치지 않았는가

## 4. 객체성

후기는 독자의 의사결정을 돕는 도구이므로, 개인적 감상보다 객관적 정보를 우선시해야 한다.

### 4.1 광고성/홍보성 표현 금지

**기준:** "강력 추천합니다", "꼭 써보세요", "최고의 선택" 같은 마케팅성 표현은 금지.

| 금지 | 이유 |
|------|------|
| "강력 추천합니다" | 근거 없이 신뢰를 요구 |
| "꼭 사용해보세요" | 독자 상황을 모름 |
| "후회하지 않습니다" | 개인의 경험을 일반화 |
| "이것만 있으면 됩니다" | 과장 광고 |

**대체:** "~한 상황이라면 고려해볼 만하다", "~한 용도에 적합하다"

### 4.2 대상 독자 명시

**기준:** 이 후기가 누구에게 도움이 될지 명시해야 한다.

**예:** "스타트업 CTO가 클라우드 비용 최적화를 고려 중이라면", "FastAPI로 첫 배포 자동화를 고민하는 주니어라면"

**검토 시 확인할 점:**
- 후기의 추천/비추천 대상을 특정 독자층으로 한정했는가
- "모든 개발자에게" 같은 포괄적 대상은 없는가 (있으면 수정)

## 5. 금지 규칙

### 5.1 금지 표현

| 금지 | 이유 | 대체 |
|------|------|------|
| "참고로" | 딴길로 새는 느낌 | 본문 통합 또는 생략 |
| "여기서 중요한 점은" | 있는 그대로 쓰면 중요함이 드러남 | 불필요한 메타 코멘트 |
| "한마디로" | 한마디로 요약되는 경험은 가치가 낮음 | 자연스럽게 서술 |
| "사실 ~다" | 강조가 아니라 변명처럼 들림 | 과감히 생략 |
| "~~에 대해 알아보자" | 클리셰 | "~~을 경험했다"로 시작 |
| "매우/정말/너무/진짜" | 강조의 반복은 효과 감소 | 구체적 수치로 대체 |
| "역시" | 예상했다는 뉘앙스, 객관성 훼손 | 그냥 사실만 |

### 5.2 객관성 훼손 금지

| 금지 | 이유 |
|------|------|
| 객관적 근거 없는 주관적 평가 | 신뢰도 하락 |
| 광고성/홍보성 표현 | 블로그의 독립성 훼손 |
| 무료 체험만으로 쓴 후기 | 깊이 부족, 사용 조건 명시 필수 |
| 한자 | 전면 금지 |
| ASCII 그림/표 | 텍스트 기반 시각 자료 금지 |

### 5.3 스포일러/결말 금지

**기준:** (교육/컨퍼런스 후기 등에서) 핵심 경험의 결말이나 결과를 상단에서 모두 알려주지 않는다. 개요에는 필요 정보만, 상세는 본문에서.

**검토 시 확인할 점:**
- 요약 정보가 지나치게 자세해서 본문을 읽을 필요가 없어지지 않았는가

## 부록: 검토 결과 판정 기준

각 기준 항목에 대해 아래 3단계로 판정한다.

| 판정 | 의미 | 조치 |
|------|------|------|
| **통과** | 기준 충족 | 수정 불필요 |
| **경고** | 기준에 미달하나 후기로서 가치는 유지 | 수정 권장, 생략 가능 |
| **실패** | 기준 미달로 독자의 의사결정 지원이라는 후기의 목적을 달성하지 못함 | 반드시 수정 후 재검토 |

**최종 판정 기준:**
- 실패가 0개이고 경고가 3개 이하 → **통과**
- 실패가 1개 이상 → **수정 필요** (모든 실패 항목 해결 후 재검토)
- 경고가 4개 이상 → **개선 권장**

### 검토 보고서 양식

```
## 검토 결과: {통과/수정 필요/개선 권장}

### 실패 항목
1. [기준명] — {구체적 위치} — {문제점}

### 경고 항목
1. [기준명] — {구체적 위치} — {문제점}

### 종합 의견
- 강점:
- 개선 필요:
- 권장 조치:
```
