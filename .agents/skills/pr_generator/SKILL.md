# PR 생성 가이드 (pr-generator)

이 스킬은 현재 작업 공간의 변경 사항(`git diff`)과 사용자-에이전트 간의 대화 맥락을 기반으로 기획 의도, 구현 결과, 그리고 기술적 비고가 논리적으로 드러나는 PR(Pull Request) 문서를 생성하기 위해 사용됩니다.

## 1. PR 문서 구조 요구사항

생성되는 PR 문서는 반드시 다음 3가지 핵심 섹션을 포함해야 합니다:

### A. PR 개요 (두괄식 구성)
* **두괄식 서술**: 문단의 첫머리에 "이 변경이 왜 발생했는지", "사용자의 어떤 핵심 요구사항을 해결했는지" 핵심 요약을 1~2문장으로 명확히 드러내야 합니다.
* **의도와 결과의 대조**: 단순한 코드 수정을 나열하는 것이 아니라, 사용자와의 대화 과정에서 도출된 기획적/기술적 의도(Why)와 그에 따른 실행 결과(What)를 명확히 대조하여 작성합니다.

### B. 주요 변경된 파일들
* 변경이 발생한 주요 파일 경로를 마크다운 링크 형식(`[파일명](file:///절대경로)`)으로 나열합니다.
* 파일별로 변경사항이 발생한 배경과 핵심적인 코드 역할(어떤 컴포넌트/로직이 추가/수정되었는지)을 간략하고 구체적인 문장으로 명세합니다.

### C. 비고
* 기술적인 제약 조건, 의존성 설치 안내, 환경 변수(`.env`) 설정 사항, 혹은 실행 및 검증 명령어를 기재합니다.
* 라이브러리 간 비동기 루프 충돌 같은 특이 사항이 있다면 해당 섹션에 작성하여 보완합니다.

## 2. 에이전트 실행 지침
1. 현재 작업 공간의 변경 사항을 명확히 파악하기 위해 **현재 브랜치와 `main` 브랜치 간의 차이(예: `git diff main..HEAD`)**를 비교 분석하여 기술적인 변경 사항을 도출합니다.
2. 기획 및 설계 단계의 **변경 의도(Why)**는 에이전트와의 대화 세션 기록을 복기하여 추출합니다.
3. 만약 대화 기록이 짧거나 불명확하여 변경 의도를 얻어내기 어려운 경우에는, 자의적으로 작성하지 말고 **사용자에게 변경 의도를 명확하게 질문**하여 확인한 후 문서를 완성합니다.
4. **문체 가이드라인 (극도의 간결성)**:
   * 문서 전체에서 불필요한 수식어나 사족을 배제하고 꼭 필요한 핵심 내용만 담백하게 포함합니다.
   * 주요 변경 사항 서술 시, 서술형 존댓말(예: "~를 구현하였습니다.") 대신 핵심 동작 위주의 **간결한 개조식/명사형 종결 문체**(예: "~ 추가", "~ 수정")를 사용합니다.
     * *나쁜 예*: 새로 설계된 파일 업로드(/upload) 및 평가 실행(/eval/run) 엔드포인트에 대한 Mocking 기반 API 통합 테스트를 구현하였습니다.
     * *좋은 예*: 신규 파일 업로드 API, 평가 API 엔드포인트에 대한 Mocking 기반 테스트 추가
5. 완성된 PR 내용은 항상 사용자가 즉시 복사하여 사용할 수 있도록 프로젝트 루트 하위의 `pr-docs/MM-dd-{PR 제목}.md` 파일로 생성합니다. (예: 06-24-RAG-다중-청킹-및-성능-평가-대시보드-구축.md)
