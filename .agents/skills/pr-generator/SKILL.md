# PR 생성 가이드 (pr-generator)

현재 작업 공간의 Git 변경 사항과 사용자-에이전트 대화 맥락을 근거로 PR 문서를 작성한다. 추측으로 변경 의도를 만들지 않는다.

## 작업 절차

1. `git status --short`, 현재 브랜치, 기본 브랜치와의 merge-base를 확인한다. 기본 브랜치가 `main`이 아니면 저장소의 기본 브랜치를 사용한다.
2. `git diff --stat`, `git diff <base>...HEAD`, 필요 시 작업 트리 diff를 읽어 실제 변경 파일과 검증 상태를 파악한다.
3. 대화 맥락에서 Why를 추출한다. 근거가 없으면 사용자가 PR 의도 또는 제목을 지정하지 않은 경우에만 짧게 질문한다.
4. 아래 형식의 Markdown을 작성한다. 사용자에게 제목을 받지 못했으면 변경 내용에서 간결한 제목을 정한다.
5. 사용자가 PR 문서 파일을 요청했거나 기본 산출물을 원하면 `pr-docs/MM-dd-{PR 제목}.md`에 저장한다. 채팅용 초안만 요청한 경우에는 파일을 만들지 않는다.

## PR 문서 형식

생성되는 PR 문서는 반드시 다음 3가지 핵심 섹션을 포함해야 합니다:

### PR 개요

- 불릿 리스트로 변경 배경·의도와 핵심 결과를 먼저 제시한다.
- 불필요한 수식이나 추상적 표현을 피하고, 결과를 검증 가능한 사실로 쓴다.

### 주요 변경 파일

- PR 플랫폼에서 동작하는 인라인 코드 경로(예: `` `app/service.py` ``)를 사용한다. 로컬 IDE 링크가 필요한 문맥에서만 절대 경로 Markdown 링크를 추가한다.
- 파일별 배경과 핵심 역할을 한 문장으로 적는다.

### 비고

- 실제로 확인한 테스트·린트 명령과 결과를 적는다. 실행하지 않은 검증은 실행한 것처럼 쓰지 않는다.
- 환경 변수, 의존성, 마이그레이션, 호환성 제약, 후속 작업처럼 리뷰어가 알아야 할 사항만 포함한다.

## 문체

- 간결한 개조식·명사형 종결을 사용한다. “~ 구현하였습니다” 대신 “~ 추가”로 쓴다.
- 한 항목에는 하나의 변경만 담고, 불확실한 내용은 사실과 추론을 구분해 쓴다.
